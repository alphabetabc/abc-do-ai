老系统怎么用 AI 改，而不把线上改炸

历史项目维护

一句话：改老代码之前，先写清楚三件事——现在实际是怎样、要改成怎样、这次不许动什么。写不清，就不许让 AI 大改。

先统一几个说法（别被缩写吓到）

说法人话
as-is现状：用户或调用方今天真实看到的行为（不是旧文档怎么写、不是你觉得应该怎样）
to-be目标态：改完之后应该变成怎样，最好和现状一条对一条
非目标这次明确不做的事。不写清楚，AI 很容易「顺便重构」
影响面会不会动到页面、接口、数据库、权限、下游、文档——逐项回答是/否
design / 契约docs/design、OpenAPI 等对外约定。代码改了行为，这些必须一起改
冻结人审过、可以按这份文档开工；开工后再改范围要重新审

三条硬规矩

  1. 没有写好并审过的「现状 → 目标态」（含影响面、非目标、验收），不要让 AI 大改老代码。
  2. 接口/表结构/可见行为已经变了，却不改 design / OpenAPI，这单不算做完。
  3. 其实是在做一整块新功能 → 别硬套本页，改走 新项目流程。
老系统最危险的不是「AI 不会写」，而是你没问清楚现状，它用猜测补全，还顺手改你没批准的地方。

1. 先分清：你在做哪一种事？

很多人把「改了一个 bug」说成「历史项目改造完成了」。这是两回事：

你在做的事人话怎么才算做完
改一个需求 / 修一个缺陷
（日常最常见）		 这一票怎么安全改完？ 有一份审过的「现状→目标态」；代码、测试、相关文档同一批合并
把整个老仓库逐步改成能稳用 AI
（技术经理排期那种）		 这个仓以后怎么持续、安全地改？ 模块列清楚了、每一块知道怎么推进、关键流程有书面现状、文档和真实行为不再对不上

日常改需求：走下面第 2 节。
领导说「把历史项目 AI Coding 化」：走第 4 节，再拆成很多次「改一个需求」。

2. 日常：改一个需求，按这 6 步做

① 查清现状→ ② 写目标态→ ③ 让 AI 挑毛病→ ④ 列出要改的文档→ ⑤ 按文档改代码→ ⑥ 对照验收合并
  1. 查清现状（as-is）：用第 3 节里的办法取证，写成「今天实际怎样」。猜的不算。
  2. 写目标态（to-be）：和现状一条对一条；写清改哪几条、这次不做哪些、影响哪些面、怎么验收算过。
  3. 先别改代码：把文档丢给模型，只让它提反向问题；你答完、补进文档，人说可以再开工。
  4. 列出文档同步项：要动接口/表/权限的，写出要改哪些 design / OpenAPI / spec。
  5. 提示词挂上这份文档：只改影响面里写的范围；非目标里写的一律不做。
  6. 对照验收打勾：代码和文档同一批进合并请求（MR）。

目标态必须写清：

  • 相对现状，到底变了哪几条
  • 这次不做的事（例如：不重构别的模块、不加没要的字段、不改无关文案）
  • 页面 / 接口 / 数据 / 权限 / 下游 / 文档 —— 每一项写「动」或「不动」
  • 验收条目要能打勾、也能判定失败（别写「体验更好」）
不写「这次不做」= 等于默许 AI 随便加戏。

3. 现状（as-is）怎么查清？一共 6 种办法

写现状之前先选办法。总共就这 6 种；可以组合，但必须写明：主要靠哪一种、辅助靠哪一种。

编号办法（人话）你具体做什么能不能单独当真
P1 自己跑一遍 按主流程点页面，或调真实/预发接口，记下输入→输出→异常情况 可以（默认首选）
P2 从测试 / 注释等旁证里找约定 有自动化测试就用；没有也常见——改看单测碎片、关键注释、TODO、示例请求/返回、fixture、断言文案。把它们整理成「代码旁白里的约定」，再拿去核实。重构前若完全没有，可先补几条锁现状的单测 旁证可靠且已核实后可当辅证；光靠注释不能定大改
P3 读代码推断 读实现和分支，写出「代码看起来会怎样」；标上「我不太有把握」 环境搭不起来时用；不要单独定大改
P4 整理旧文档（可让 AI 归纳） 旧 wiki/spec 常对不齐：把散落材料丢给 AI 总结、去重、合并成一版「候选现状」，再和真实行为逐条对（一致 / 过期 / 冲突）。对得上的可回写成新版 design；对不上的标过期，别当真相 不能单独当真（归纳稿仍是草稿，必须用 P1/P3 核实）
P5 问现场的人 没有工单系统也没关系：问现场运维、产品经理、熟手研发——「线上一直怎么用的、哪些坑不能动」。先写成「待验证」，再自己跑一遍或读代码核实 不能单独当真
P6 看线上证据 日志、审计、网关 trace、浏览器 HAR、监控、错误码分布；写清查的时间段和条件 偶现/只有生产才有时很有用

怎么选？看你的处境

你现在的情况主要用再补
本地或预发能点出来P1 自己跑一遍P4 看旧文档是否过期
有靠谱测试，或至少有关键注释/示例可整理P2 整理旁证（无测就用注释等）P1 抽查主流程；要重构时尽量先补锁现状的单测
环境搭不起来，只能看代码P3 读代码P5 问运维/产品；有条件再看 P6 线上
旧文档又多又对不齐P4 先让 AI 归纳成一版候选必须再加 P1 或 P3 核实；核实后可回写成新版文档
「运维/产品说以前就这样」P5 先记下来必须再用 P1 或 P3 核实
偶现、只有生产才出现P6 线上证据能复现再补 P1

记住两句：

  • 能点页面 / 调接口 → 优先自己跑（P1）。
  • 旧文档、运维/产品口头说法只能当线索，不能当唯一真相。
  • 屎山没有自动化测试很正常：P2 改看注释/示例/零星单测；别因为「没测试」就跳过旁证。

现状写合格的最低标准：写的是今天能观察到的行为；写了证据来源；写了主流程和已知边角；写了「本期先不修的老坑」（防止 AI 顺手「修好」）。

4. 整仓改造：技术经理怎么推进？

目标不是一天重写完，而是：以后每次改都能按纪律做，改完后文档和真实行为对得上。

按顺序做这 5 步：

  1. 列清单：有哪些模块、主流程、对外接口；现有文档还信不信。
  2. 给每块贴标签（只贴一个主标签）：要改行为 / 修缺陷 / 重构但对外不变 / 其实是新功能 / 不再维护。
  3. 选推进方式（下面 5 选，可组合）。
  4. 先把关键主流程的现状写下来（用第 3 节的办法）。主流程都写不清，不许开大改。
  5. 拆成小批：每一批按第 2 节「改一个需求」做完；合入时文档一起改。

整仓推进方式一共 5 种（S1–S5）

编号叫什么什么时候用你具体怎么做
S1 来一个需求改一个 平时修修补补，没有整仓计划 每次严格走第 2 节；合并时必须改相关文档，否则文档债越堆越多
S2 按模块一块块啃 要系统改造某一个业务域（比如账号、订单） 先列该域清单 → 写清主流程现状 → 再一批批改目标态
S3 先把真实行为写进文档 代码很乱，但对外接口还相对稳定；或大改之前 先按真实行为改 design / OpenAPI（可以开「只改文档」的合并），再动代码
S4 先用测试锁住，再动内部 要重构、换实现、升级依赖，但对外行为暂时不变 用测试/验收表锁住现状 → 再重构 → 测试保持全绿；禁止借重构偷偷改行为
S5 按新项目流程做 其实是新模块；或现状根本写不清；或跨很多域要单独排期 停用本页模板，走 新项目流程（产品输入 + 五件套）

怎么选（别想复杂）：

  • 平时交付 → S1（但每张合并请求都要改文档）
  • 某个业务域要系统化搞好 → S2 + S3（先写清现状和接口文档，再一块块改）
  • 大重构、对外先不变 → S4；若行为也要变，拆成「先锁旧行为，再单独开一单改行为」
  • 新功能 → 只走 S5，别假装是维护

一批怎么切：先主流程后边角；先接口/权限/数据，后纯文案；一批只做一个能演示的目标（例如「列表筛选和接口约定对齐」），不要「本周重构整个账号域」。

某个模块什么时候算「改造阶段完成」：不是「用过 AI 了」，而是——清单和标签齐了；主流程有可引用的现状说明；后来的变更都能拿出「现状→目标态」；对外行为和 design 没有已知对不上的地方。

整仓可以一直「进行中」。按模块勾完成即可，不必等整个仓库重写完。

5. 什么时候别用本页，改走新项目？

  • 要新增一整块用户能感知的能力（不只是改旧行为）
  • 要新的主数据模型，或一批新接口（不是改几个字段）
  • 跨很多业务域，需要单独里程碑
  • 现状根本写不清——应先调研或补规格，不要硬写目标态

6. 改代码前：让模型只提问（必做）

模板填完先别改代码。把文档贴给模型，只让它找漏洞；你答完写回文档，人确认后再开工。

你是苛刻的遗留系统变更审稿人。请只基于我粘贴的「现状 → 目标态」文档提问。
要求：
1) 检查：现状证据够不够（P1–P6）、能不能复现、目标态能不能观察、
   一对一对照清不清、非目标能不能挡住顺手重构、
   影响面有没有漏接口/数据/下游/权限、兼容与回滚、验收能不能判定失败；
2) 每个问题说明：不问清楚时 AI 最可能改错什么；
3) 区分：必须本期拍板 / 可以以后再定；
4) 禁止直接改代码或擅自扩大范围；不知道就提问。

7. 改代码时怎么约束 AI

改了接口字段 → 改 api 文档和 OpenAPI；改了表 → 改 data-models；原来有五件套 → 同步验收场景。禁止：代码已合并，文档还在描述旧行为。

请严格依据本「现状 → 目标态」文档实现变更。
只修改「影响面」标明的范围；「非目标」里的一律不要做。
不要引入文档没写的字段、接口、菜单或行为。
若现状描述与代码不一致，先停下来说明差异，不要猜测后继续大改。
若要改接口约定或表结构，先列出要同步的 docs/design（或 OpenAPI），再改代码。
完成后对照「验收对照」逐条说明怎么验证。

8. 文档写在哪？

写什么放哪
这一票的现状→目标态相关特性目录下的变更说明，或合并请求描述附件（团队统一一种即可）
接口 / 表结构变更docs/design、OpenAPI、data-models
原来就有五件套的行为同步改该特性的 spec / acceptance-tests
探查过程的笔记可先放 docs/research/；一旦写进正式约定，必须回写 design

完整可复制模板见同目录 Markdown：legacy-project.md 第 10 节（含「证据用了 P几」必填项）。

9. 别这么干

  • 「就改一小下」→ 不写文档，直接让 AI 扫仓库改
  • 现状只抄旧文档、只信 AI 归纳稿，或只听运维/产品口头说——自己不跑、不核实
  • 因为「没有自动化测试 / 没有工单」就空着 P2、P5 不用
  • 目标态只写「优化体验 / 更合理」，没法验收
  • 不写「这次不做」，等于默许重构和加戏
  • 接口/表改了，design / OpenAPI 不动
  • 明明是新功能，硬塞进本模板，逃避产品输入和五件套
  • 宣称「整仓改造完成」，但主流程没有书面现状，文档和真实行为还对不上
