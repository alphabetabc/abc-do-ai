# 为什么做与思维转变

> 本页讲清一件事：为什么现在必须推 AI Coding，以及个人与团队要改的底层逻辑。  
> 具体开发流程见后续文档。

---

## 开篇：AI 出现是好是坏？

**一句话答案：对还在用旧能力结构吃饭的人是坏消息；对愿意重定义问题、用 AI 放大闭环的人是好消息——中立观望等于选坏的那一侧。**

不是讨论「AI 酷不酷」，而是：行业成本曲线已经变了，你站哪边。

---

## 1. 培训目的

拉齐全员对 AI Coding 的认知，并约定同一套工作方式。

要达成三点共识：

1. **为什么推**——不是尝鲜，而是经营与能力结构倒逼；**紧迫感不足就已经危急存亡**  
2. **核心能力是什么**——不是「会调模型」，而是**会把问题问清楚**  
3. **个人与团队怎么变**——个人：单兵闭环、抛弃存量、产品思维、不断更新；团队：编制更瘦、壁垒更少、文档可接  

工具操作与产品业务规格不在本篇展开。

**1）真正变了什么**

- 传统编程：问题多半在脑子里 → 人手写实现 → 人验收（规格常隐性、事后补文档）。  
- AI Coding：人把问题写成**可执行文档** → AI 按文档实现 → 人对照验收把关。  
- 变的不是「谁敲键盘」，而是**问题定义必须外置、可传递、可验收**——否则 AI 只会加速做错。

**2）为什么不是每个人**

工具人人有；多数人只是「用 AI 辅助传统编程」——聊天补全、看着像就收、没有冻结规格、没有「本次不做」、没有验收对照。  
瓶颈在**会不会问**，所以打开模型 ≠ 在做 AI Coding。

**3）重要指标（可对照自检）**

1. 动手前有可分享、已写清的文档（新项目：冻结 PM 输入/五件套；历史：现状→目标态）  
2. 提示词**挂文档路径**，不是只贴聊天记录  
3. 有可勾选、可判定失败的验收；合并前对照打勾  
4. 契约/行为变了，文档与代码同批更新  
5. 文档不清或 AI 越界时会**停下来问**，而不是猜完继续大改  

五条里多数做不到 → 你只是在「用 AI 写代码」，还不是 AI Coding。

## 2. 为什么现在推：公司与团队现状

我们是一个**偏前端 + UI** 的团队。过去强项在体验与界面；后端、联调、端到端闭环往往靠「多人串联」补齐。

这句话里的**「紧日子」**，不是口号，而是具体处境：

| 「紧日子」指什么 | 落到我们身上 |
|------------------|--------------|
| 项目更挑、预算更敏感 | 纯「做界面交付」的空间收窄，客户更敢砍价、更敢换供应商 |
| 盈亏与编制被盯紧 | 同样产出若仍靠堆人头、堆会议、堆手工联调，**边际成本买不起** |
| 交付节奏被压缩 | 「讲需求 → 做页面 → 等接口 → 再联调」这条多人串联链，同步成本越来越高、周转越来越慢 |

所以：**不是「最好也学学 AI」——而是旧路径在紧日子里已经越来越贵、越来越慢；还不改，就是危急存亡，不是优化项。**

### 2.1 外部与经营压力（为什么说危急存亡）

- **外部环境偏紧**：项目更挑、预算更敏感，纯界面交付空间在收窄  
- **盈亏压力现实存在**：堆人头、堆会议、堆手工联调，利润被同步成本吃掉  
- **客户要的是结果**：从诉求到可上线能力的闭环，而不是「页面做完了再等后端」  

多人串联的**同步成本**，正在变成利润的直接敌人。  
还把 AI Coding 当成「尝鲜 / 加分项」的团队，会在下一轮比拼里直接掉队。

### 2.2 能力结构必须升级：瓶颈已经换位

AI 把「实现带宽」大幅交给个人之后，真正拉开差距的不再是「会不会写」，而是「会不会问」。

| 过去的瓶颈 | 现在的瓶颈 |
|------------|------------|
| 我 / 我们会不会实现（页面、接口、联调） | 我 / 我们能不能把问题定义到可执行、可验收 |
| 差在手速与熟练度 | 差在提问质量与边界清晰度 |

**不是 AI 不能帮你实现，而是你根本不知道如何去问。**

问不清楚时，AI 特别擅长产出「看起来完整的错误系统」——字段臆造、范围漂移、验收对不上，返工成本被放大。  
问得清楚时，同一套人可以少开会、少等人，把一条薄的端到端链路自己跑完。

#### 什么叫「好问题」

好问题不是「帮我做漂亮一点」，而是让执行方（人或 AI）无需再猜：

- **给谁用、解决什么痛**（背景）  
- **做完长什么样、怎样算对**（验收）  
- **明确不做什么**（非目标）  
- **约束是什么**（权限、数据、兼容、范围）  
- **现状是什么 → 目标是什么**（尤其历史系统）  

后续文档里的 PM 输入、spec 五件套、as-is→to-be，本质都是把「好问题」固化成可传递的工件——不是为了写文档而写文档，而是为了让 AI 与异步协作变得可用。

#### 为什么偏前端团队更痛、也更该先转

- 过去「链路不全」可以用协作掩盖；现在协作变贵  
- AI 能补实现缺口，但**补不了你脑子里没成型的问题**  
- **个人产品规划能力** = 把模糊诉求收成好问题的能力；这才是全栈闭环的真正门槛  

### 2.3 一句话结论

在成本与环境压力下，团队要的不是「更多会敲提示词的人」，而是「更能把问题问清楚、并能单兵闭环」的人。  
AI Coding 是手段：**规格与验收进文档 → AI 加速实现 → 人负责判断与把关。**  
**不转，不是慢半拍——是旧能力结构在紧日子里活不下去。**

## 3. 思维转变

### 3.1 个人视角：单兵作战 · 抛弃存量 · 产品思维 · 不断更新

深层逻辑：

1. **单兵作战**：一个人要能完成「定义问题 → 驱动 AI 实现 → 对照验收」。这是 AI 时代的基本盘。  

2. **抛弃存量**：别再把「我会某套框架 / 某套组件 / 某段老项目经验」当护城河。  
   - **存量** = 过去攒下的手活熟练度、口头里的门道、只存在某个人脑子里的上下文。  
   - 这些东西折旧极快：AI 正在摊平「会写」的优势；只靠存量吃饭，会在下一轮被会提问、会更新的人超过。  
   - 抛弃不是否定经验，而是**停止依赖存量当安全感**——经验要变成「更好的问题」和「可进仓库的文档」，而不是「只有我懂」。  

3. **产品思维**：不只「把分配到的页面做完」，而是能从用户与业务侧把问题问清楚。  
   - 想清楚：给谁用、解决什么痛、怎样算做成、本期明确不做什么。  
   - 会排优先级与验收口径，而不是等产品把每一步喂到嘴边。  
   - 偏前端尤其要补：把模糊诉求收成好问题的能力，就是个人产品规划能力——这才是全栈闭环的真正门槛。  

4. **减少协作**：减少为对齐信息而开的会、等某人口头里的字段、等联调窗口；保留真正的边界决策。  

5. **在信息不对称下仍能快速交付**（两层都要）：  
   - **对内**：关键上下文进文档后，不再依赖「只有某某知道」；信息差被抹平，速度来自少等待。  
   - **对外 / 对项目**：现场信息往往不齐、不同步；谁能更快把残缺信息收成「可问的问题 + 可验的范围」，谁就能先交出一版可验收结果，而不是卡在「等所有人对齐」。  

6. **不断更新**：单兵能走多远，取决于你脑子里「能问出什么问题」。技能栈与视野不更新，就只会用旧问题套 AI，产出也旧。要持续补齐并刷新，例如：  
   - 全栈技术（不只停留在页面层）  
   - 性能瓶颈（会定位、会取舍，而不是只会「看着卡」）  
   - 产品与体验（范围、优先级、验收——与上条「产品思维」互相加强）  
   - 智能体开发（Agent / 工具 / 编排等新范式）  
   - 前沿解决方案与竞品分析（知道行业怎么解题，才问得出更好的题）  

**速度来自「我能定义并闭环」，而不是「我把所有人都拉进会」。**  
**杠杆来自「我不断更新能问的问题」，而不是「我只会熟练地问旧问题」。**  
**安全感来自「我能抛弃存量再站起来」，而不是「我手上还有几年前的熟练度」。**

| 以前常见 | 现在默认 |
|----------|----------|
| 先问人、等人口头讲清再动手 | 先写出可问的问题结构，再带着缺口去确认 |
| 协作越多越安心 | 协作过密往往说明问题还没被定义清楚 |
| AI 当聊天补全，看着像就收 | AI 当「按文档执行的手数」；对照验收挑错 |
| 价值 = 做完分配给我的那一块 | 价值 = 我能独立端到端交付一个薄片 |
| 等产品喂需求，自己只做实现 | **产品思维**：能把模糊诉求收成可验收问题 |
| 护城河 = 熟悉某套组件 / 交互细节（存量） | 护城河 = 好问题能力 + 产品思维 + 持续更新 + AI 放大半径 |
| 学会一套技术吃几年 | **抛弃存量依赖**；主动更新：全栈、性能、产品、智能体、前沿方案与竞品 |
| 知识只在我脑子里才有价值 | 知识进文档才有复利；只在脑子里的是风险 |

个人习惯：

1. **先写清再生成**——没写清行为，不让 AI 大改  
2. **我审文档与验收，AI 写实现**——责任边界清楚  
3. **每次交付留下可复用文档**——抹平对内信息差，也方便下周的自己  
4. **定期更新自己的问题库**——用新认知去问更好的题，而不是用 AI 重复旧解法  
5. **定期清点并丢掉「存量安全感」**——问自己：离开这套旧熟练度，我还能不能定义问题并闭环？  
6. **动手前先过一遍产品问题**——给谁用、痛点、验收、非目标，写不出来就先问、先补文档  

### 3.2 团队视角：编制更瘦 · 壁垒更少 · 文档可接

口径要硬、要清楚——团队要同时做到三件事：

**A. 编制更瘦**  
同等产出所需编制下降；编制不降，则同等编制必须扛更高产出与更完整闭环。  
减的是靠「多人串联、天天同步」堆出来的冗余，不是鼓励无人负责。

**B. 壁垒更少**  
有人把关键逻辑、接口约定、部署门道、历史坑只留在自己脑子里或私聊里，用「只有我懂」换不可替代感。  
这在紧日子里更致命：人一走系统就停；人在时别人也无法单兵闭环，AI 也问不出真相。  
**团队默认立场：壁垒不是能力，是组织风险；不靠卡脖子。**

**C. 文档可接**  
交付完成的标准不是「代码合进去了」，而是**换一个人（或下周的你，或 AI）按仓库文档就能接着干**。  
契约、行为、验收、关键坑进 design/spec；MR 文档与代码同批；问不到文档 = 流程失败，不是「再找某某问一声」。

深层逻辑：

1. **减人增效**  
   过去用多角色接力堆产能；现在用「更少的人 × AI × 好问题」替代。人头花在同步上，不如花在定义问题上。  

2. **拆技术壁垒，禁止靠信息不对称保饭碗**  
   - 表现：口头才给字段、文档故意不写全、关键路径「找我」、评审用黑话把门、拒绝把约定写进 design。  
   - 后果：编制降不下来、单兵闭环做不到、AI Coding 在本团队名存实亡。  
   - 对策：契约与关键行为必须进仓库文档；合并要求文档同批；「只有某某知道」本身就是缺陷，要开单还清。  

3. **文档可接 = 组织记忆落地**  
   不在编制、不在个人口袋。人少时更怕知识只在某个人脑子里；**可接的文档**是小团队存活条件，也是拆壁垒的硬手段。  

4. **统一「如何提问」= 统一质量下限**  
   模板是团队级提问标准，避免只有个别「会问的人」能用 AI。  

5. **默认异步 + 文档，而不是默认同步**  
   会议变短、变少；结论必须回写。人少也能转；壁垒也无处藏身。  

| 以前常见 | 现在默认 |
|----------|----------|
| 对齐靠会议与口头，文档事后补 | 对齐靠可执行文档；会议结论必须回写 |
| 接口字段聊着定，MR 对不上 | 契约与文档同批更新；禁止臆造 |
| 「关键逻辑在我这，你们问我」 | 关键逻辑进 design/spec；问不到文档 = 流程失败 |
| 用黑话 / 私有门道卡协作 | 用统一模板与公开契约协作；壁垒要被拆掉 |
| 代码合了就算交付 | **文档可接**才算交付：别人按文档能接着干 |
| 少数「会 AI 的人」单兵突击 | 全员同一套提问与交付标准 |
| 知识在某个人脑子里 | 知识在仓库文档里；换人仍可继续 |
| 编制靠分工串联撑产出 | 编制靠单兵闭环 + AI 杠杆撑产出 |

团队三条铁律：

1. 没有写清的行为，不让 AI 大改  
2. 契约 / 验收变了，文档与代码同批更新（或紧跟随，团队统一）  
3. 禁止臆造：文档里没有的字段、接口、菜单，不要顺手加  

说明：减少的是**噪音协作**，不是禁止沟通。跨边界争议、破坏性变更、验收口径冲突，仍要同步——但结论写回文档。

### 3.3 同一套尺子（个人 ↔ 团队）

| | 尺子 |
|--|------|
| **个人价值** | 会提问 + 能闭环 + 肯更新（含产品思维、抛弃存量） |
| **团队价值** | 编制更瘦 + 壁垒更少 + 文档可接 |

两边必须对齐：个人只会「会写」却不沉淀文档 → 团队「文档可接」落空；团队仍靠壁垒与堆人 → 个人「单兵闭环」无处发力。  
**用同一套尺子衡量人与组织，AI Coding 才站得住。**

## 4. 团队发展方向（三个应用）

这是团队要去**争取做成的几件核心事**。AI Coding 是底座能力；三条方向是业务主战场。本篇只定调，不展开规格。

| 方向 | 定位（我们为什么要抢） |
|------|------------------------|
| 数字孪生 × AI 智能体（结合大屏） | **发挥前端的最后优势**：复杂可视化、态势呈现、人机交互仍是我们最硬的阵地；叠上智能体，把「看得见」升级成「可行动」 |
| AI 智能体平台（Hermes / OpenClaw，类 WorkBuddy） | **最前沿的智能体开发模式**：站到 Agent 运行时、工具与治理的主航道上，而不是只做页面外壳 |
| AI 产品研发平台（类 Coze） | **AI Coding 的最终方向——授人以渔**：不只自己会用 AI 交付，还要把「提问 → 生成 → 验收」产品化，让更多人能搭、能改、能闭环 |

三者递进关系可以记成：

1. **用前端优势拿下一仗**（孪生 × 智能体 × 大屏）  
2. **用前沿范式建立智能体平台能力**（Hermes / OpenClaw，类 WorkBuddy）  
3. **把能力沉淀成可复制的产品研发平台**（类 Coze，授人以渔）  

**底座（会提问、产品思维、抛弃存量、单兵闭环、编制更瘦、壁垒更少、文档可接、不断更新）不到位，这三仗都打不稳。**
