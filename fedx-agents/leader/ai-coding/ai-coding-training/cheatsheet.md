# 提示词与自检

> 日常打开本页**复制提示词**即可。  
> 流程细节回看：[思维转变](./mindset.md) · [新项目](./new-project.md) · [历史项目](./legacy-project.md)

---

## 0. 先选路（30 秒）

| 你在做什么 | 走哪条 | 动手前必须有什么 | 核心文档 |
|------------|--------|------------------|----------|
| 新能力 / 新模块 / 要从零写规格 | **新项目** | 产品审定冻结的 PM 输入（含反向提问） | PM 输入 → 五件套 → design |
| 改老系统已有行为 / 修缺陷里的行为变更 | **历史项目** | 审过的「现状 → 目标态」（含影响面、本次不做、验收） | as-is → to-be → 回写 design |
| 其实是新功能，却想套历史模板 | **改走新项目** | — | 不要硬套历史模板 |
| 现状根本写不清 | **先调研 / 补规格**，或升级新项目 | — | 不要硬写目标态开改 |

**三条铁律（新项目、历史项目都适用）：**

1. 没有写清的行为，不让 AI 大改  
2. 契约 / 验收变了，文档与代码同批更新  
3. 禁止臆造：文档里没有的字段、接口、菜单，不要顺手加

## 1. 数字速查（讲完课还记得住）

### 1.1 新项目主路径

```text
背景 → 技术确认 → assets → PM 输入草稿
  → ★ 反向提问并冻结 PM 输入
  → 五件套 + 同步 design（按需 research）
  → 按五件套实现（能测则先红后绿）→ 验收同批
```

五件套：`spec.md` · `plan.md` · `tasks.md` · `data-model-extensions.md` · `acceptance-tests.md`

### 1.2 历史：改一个需求 6 步

```text
查清现状 → 写目标态 → ★ 让 AI 挑毛病并冻结
  → 列出要改的文档 → 按文档改代码 → 验收并和文档一起合并
```

### 1.3 现状怎么查清（P1–P6）

| # | 人话 | 能不能单独当真 |
|---|------|----------------|
| P1 | 自己跑一遍（点页面 / 调接口） | 可以（默认首选） |
| P2 | 测试 · 注释等旁证（没自动化测试就看注释/示例/零星单测） | 核实后可当辅证 |
| P3 | 读代码推断 | 不要单独定大改 |
| P4 | 整理旧文档（可让 AI 归纳合并） | **不能**（归纳稿仍须核实） |
| P5 | 问现场的人（运维 / 产品 / 熟手研发；没有工单也行） | **不能** |
| P6 | 看线上证据（日志 / HAR / trace） | 偶现时很有用 |

### 1.4 整仓怎么推进（S1–S5）

| # | 人话 |
|---|------|
| S1 | 来一个需求改一个（每张 MR 强制改文档） |
| S2 | 按模块一块块啃 |
| S3 | 先把真实行为写进 design / OpenAPI，再动代码 |
| S4 | 先用测试锁住现状，再重构内部 |
| S5 | 按新项目流程做（等价新模块 / 现状写不清） |

## 2. 新项目 — 提示词（按阶段复制）

### 2.1 PM 输入写完后 — 反向提问（先于五件套）★

```text
你是苛刻的产品+研发双审。请只基于我粘贴的《产品经理需求输入》全文（及所述 assets），
列出反向问题，按优先级排序。要求：
1) 覆盖：范围边界、角色权限、主旅程分支、空态/错态、数据口径、验收可测性、非功能、与外部系统边界；
2) 每个问题说明：若不问清楚，实现阶段最可能出现什么错误臆造；
3) 区分：必须本期拍板 / 可挂起为开放问题；
4) 禁止直接写 spec 五件套或代码；禁止编造业务事实——未知就提问。
输出：问题清单 + 建议回写到文档的哪一节。
```

通过再进五件套：至少一轮有记录的问答；阻塞实现的未知已关闭或显式挂起；产品确认可据本文档生成五件套。

### 2.2 从已冻结的 PM 输入生成五件套

```text
请根据 {路径}/pm-inputs/pm-requirements-input.md（及同目录 assets），
按仓库 docs/specs/_template 在 {路径}/ 生成五件套：
spec.md、plan.md、tasks.md、data-model-extensions.md、acceptance-tests.md。
全局权威引用 docs/design/ 下既有文档与 OpenAPI；
不要编造已存在于 design 的字段定义；未知处写入「开放问题」与 plan 前置条件。
若需新增/变更 API 或表结构，列出必须同步修改的 design 文件清单，先说明再写入草稿。
```

生成后建议再对五件套跑半轮反向提问（与 PM 不一致、验收不可测、契约缺口），研发审定后再实现。

### 2.3 按五件套实现

```text
请严格依据 {路径}/spec.md 与 {路径}/acceptance-tests.md 实现。
设计契约以 {契约路径} 为准（docs/design 与 OpenAPI）。
不要引入文档中未出现的响应字段、菜单或权限码。
「非目标 / 本期不做」禁止实现。
若需变更契约，先说明变更点并更新文档，再改代码。
实现节奏：能测则先写失败用例（红）→ 最小实现（绿）→ 对照验收勾选。
完成后说明：改了哪些文件、如何对照 acceptance-tests 验证。
```

### 2.4 补验收自动化

```text
请根据 {路径}/acceptance-tests.md 中的场景，在现有测试框架下补充自动化
（或可重复的手工验收步骤记录）。
优先覆盖关键路径与契约边界；不要为尚未写入验收文档的行为编写测试。
```

### 2.5 新项目开工前打勾

- [ ] PM 输入已反向提问并产品冻结  
- [ ] 五件套已审定；acceptance 可观察、可失败  
- [ ] 要改的 design / OpenAPI / 路由已列出或已改  
- [ ] research 若有采纳项，已回写 design  
- [ ] 实现提示词指向五件套路径与非目标

## 3. 历史项目 — 提示词（按阶段复制）

### 3.1 协助查清现状（选 P1–P6）

```text
我在为遗留变更写现状（as-is）。请按取证办法帮我整理，不要编造未观察到的行为。
已知：【粘贴现象 / 链接 / 文件路径 / 日志摘要 / 旧文档片段】
请输出：
1) 建议主要办法与辅助办法（仅从 P1 自己跑一遍 / P2 测试·注释等旁证 / P3 读代码 /
   P4 整理旧文档可 AI 归纳 / P5 问运维或产品 / P6 看线上证据 中选）；
2) 一份可写入模板的现状草稿（主流程 + 边角 + 本期先不修的坑）；
3) 仍未知、必须用人复现或问运维/产品的清单；
4) 禁止假设线上行为；不知道就标【未知】。
```

### 3.2 旧文档很乱 — 先归纳再核实（P4 辅助）

```text
下面是散落的旧文档 / wiki / 注释摘录（可能互相矛盾）。
请总结、去重，合并成一版「候选现状」草稿，并标明：
1) 各来源一致之处；
2) 互相冲突之处（列出冲突点，不要擅自挑边）；
3) 完全缺失、必须靠复现或问人才能补的缺口。
禁止把归纳结果说成已核实的线上真相；输出末尾加一句：须用 P1/P3 核实后再冻结。
```

### 3.3 现状→目标态写完后 — 反向提问（先于改代码）★

```text
你是苛刻的遗留系统变更审稿人。请只基于我粘贴的「现状 → 目标态」文档，列出反向问题。要求：
1) 覆盖：现状证据（P1–P6）是否足够、能否复现、目标态能否观察、一对一对照是否清晰、
   「本次不做」能否挡住顺手重构、影响面是否漏接口/数据/下游/权限、
   兼容与回滚、验收能否判定失败；
2) 每个问题说明：不问清楚时 AI 最可能误改什么；
3) 区分：必须本期拍板 / 可挂起；
4) 禁止直接改代码或扩大目标态范围；未知就提问。
输出：问题清单 + 建议回写到模板哪一节。
```

### 3.4 挂文档实现

```text
请严格依据 {现状-目标态文件路径} 实现变更。
只修改文档「影响面」标明的范围；「非目标 / 本次不做」禁止实现。
不要引入文档未描述的字段、接口、菜单或行为。
若现状描述与代码不符，先停下来说明差异，不要猜测后继续大改。
若需改契约或表结构，先说明并列出要同步的 docs/design（或 OpenAPI）文件，再改代码。
完成后对照文档「验收对照」逐条说明验证方式。
```

### 3.5 历史变更开工前打勾

- [ ] 已判定走历史路径（不是该升级新项目的 S5）  
- [ ] 现状写明证据用了哪些 P；没把握未硬开大改  
- [ ] 目标态 / 影响面 / 本次不做 / 验收已写清  
- [ ] 已反向提问并回写；文档已冻结  
- [ ] design / OpenAPI / spec 同步项已列出或已改  
- [ ] 实现提示词挂了这份文档  

完整模板见 [历史项目维护](./legacy-project.md) §10。

## 4. 通用 — 随时可加的约束句

### 4.1 收紧范围

```text
本次任务范围仅限：{一句话范围}。
明确不做：{非目标列表}。
不要重构无关模块；不要「顺便优化」未点名的文件。
有疑问先问，不要自行扩展范围。
```

### 4.2 强迫文档同步

```text
若本次变更改变了对外 API、持久化或用户可见行为，列出需同步的文档清单
（spec / acceptance-tests / design / OpenAPI），并说明每项改了什么。
禁止：契约已变但仅更新代码。
```

### 4.3 发现未知就停

```text
遇到文档未写清的行为或字段：先停下来提问，列出未知点，
不要用「业界最佳实践」或猜测补全后继续大改。
```

## 5. 反例（不要这样写）

| 烂提示词 | 问题 |
|----------|------|
| 「帮我把这块做漂亮一点，参考业界最佳实践，你看着改就行。」 | 无权威文档、无边界、默许臆造与大改 |
| 「顺便把相关模块也重构一下。」 | 扩大非目标；历史仓极易改炸 |
| 「按旧 wiki 实现就行。」 | 裸 P4；文档常与线上不符 |
| 「运维说一直这样，直接按他说的改。」 | 裸 P5；口头须核实 |
| 只贴聊天记录、不挂仓库文档路径 | AI 下次会话丢失权威；也无法评审 |

**好习惯一句：** 提示词里写上文档路径 + 非目标 + 契约以哪份 design/OpenAPI 为准。

---

## 6. 禁止事项（全员）

- 臆造文档中不存在的字段 / 菜单 / 权限码  
- 契约已变却只改代码  
- 口头改需求不回写文档  
- 跳过验收「先合并再说」  
- 无「现状→目标态」就对遗留模块大范围自动重构  
- 无审定 PM 输入就开写新特性五件套  
- 因为「没有自动化测试 / 没有工单」就空着 P2、P5 不用  
- 把 AI 归纳的旧文档稿或人口头说法直接当冻结真相  
- 借 Frontend Design「顺便重做整站视觉」（除非目标态白纸黑字写了）

## 7. 人与 AI 默认分工（对照用）

| 谁 | 负责 | 默认不负责 |
|----|------|------------|
| 人（产品/研发/技术经理） | 冻结文档、拍板未知、审 MR、勾验收 | 把「现状真相」甩给模型猜 |
| AI | 反向提问、起草文档/代码、按文档最小 diff | 扩大范围、编造字段、当线上行为真相源 |

个人习惯（思维篇）：先写清再生成；产品思维过一遍；我审文档与验收，AI 写实现；留下可复用文档；抛弃存量；定期更新问题库。  
尺子：个人 = 会提问 + 能闭环 + 肯更新；团队 = 编制更瘦 + 壁垒更少 + 文档可接。

## 8. 「算会用」自检

### 8.1 认知题

1. 和传统编程比，真正变了什么？怎样才算在做 AI Coding？（五条指标，见思维篇开放问题）  
2. 新项目与历史项目各走哪条？动手前核心文档分别是什么？  
3. 现状查清有哪 6 种办法？哪两种不能单独当真？  
4. 整仓推进有哪 5 种方式？什么时候必须升级新项目（S5）？  
5. 人与 AI 默认分工？三条铁律是什么？  
6. 三个产品方向与本培训的关系？（愿景 vs 技能底座）  

### 8.2 实践（一周内完成任一）

**A 新项目：** 小特性 → 极简 PM 输入（含一轮反向提问）→ 五件套或直接挂文档实现 → MR 贴文档链接  

**B 历史项目：** 真实小改动 → 填完历史模板（写明用了哪些 P）→ 反向提问 → 按文档改完并勾验收  

### 8.3 勾选

- [ ] 能独立选择新项目或历史项目路径  
- [ ] 动手前有可分享、已冻结的文档  
- [ ] 提示词指向文档路径，不是只贴聊天  
- [ ] 合并前对照过验收；契约变更已回写 design
