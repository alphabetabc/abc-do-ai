# 历史项目维护（AI Coding 化改造）

> 一句话：改老代码前，先写清三件事——**现在实际是怎样**、**要改成怎样**、**这次不许动什么**。  
> 前置：[新项目开发流程](./new-project.md) · 思维底座：[为什么做与思维转变](./mindset.md)  
> 浏览器审阅版（已用人话重写）：[html/legacy-project.html](./html/legacy-project.html)

---

## 0. 先统一说法（别被缩写吓到）

| 说法 | 人话 |
|------|------|
| **as-is** | **现状**：用户/调用方今天真实看到的行为（不是旧文档怎么写） |
| **to-be** | **目标态**：改完应该怎样；最好和现状一条对一条 |
| **非目标** | 这次明确不做的事；不写清，AI 容易「顺便重构」 |
| **影响面** | 会不会动页面、接口、库表、权限、下游、文档——逐项答是/否 |
| **design / 契约** | `docs/design`、OpenAPI 等对外约定；行为变了必须一起改 |
| **冻结** | 人审过、可以按这份文档开工；开工后再改范围要重新审 |
| **P1–P6** | 查清现状的 **6 种办法**（见 §4） |
| **S1–S5** | 整仓推进的 **5 种方式**（见 §3） |

老系统真相往往不在文档里，而在代码、口头和线上行为。  
AI 上下文不全时会用猜测补全，还顺手改你没批准的地方。  
所以：**先写成可提问、可验收的「现状 → 目标态」，再允许 AI 动手。**

| 若跳过… | 典型后果 |
|----------|----------|
| 跳过查清现状 | 改的是「想象中的现状」，上线后边角全炸 |
| 跳过「本次不做」 | AI 「顺便优化」半个模块 |
| 跳过影响面 | 接口 / 下游 / 权限被静默改掉 |
| 跳过让模型挑毛病 | 文档写了但仍充满未知；实现期臆造 |
| 跳过改 design | 代码已变，对外约定仍是旧的 |

**三条硬规矩：**

1. 没有写清并审过的 **现状 → 目标态**（含影响面、非目标、验收），不让 AI 大改老代码。  
2. 行为 / 接口约定已变却不回写 `docs/design` 或相关 spec，这单不算做完。  
3. 其实是一整块新功能时，**升级走新项目流程**，不要硬套本模板。

## 1. 何时用本页 / 何时改走新项目

### 1.1 用本页

- 改旧行为、缺陷里的行为变更  
- 规格散落在代码与口头中，短期写不齐完整五件套  
- 影响面圈得住、验收能写成对照表的增量改造  

### 1.2 改走新项目

出现任一情况，走 [新项目开发流程](./new-project.md)：

- 新增一整块用户能感知的能力（不只是改旧行为）  
- 要新的主数据模型 / 一批新 API，而非局部字段  
- 跨很多业务域、需要独立里程碑  
- 现状本身写不清——应先调研 / 补规格，不要硬写目标态  

### 1.3 文档写在哪

总树见 [新项目开发流程 §1](./new-project.md)。常见落点：

| 写什么 | 放哪 |
|--------|------|
| 现状 → 目标态正文 | `docs/specs/<相关特性>/` 变更说明，或 MR 附件（团队统一一种） |
| 接口 / 表结构变更 | 回写 `docs/design/api*`、OpenAPI、`data-models.md` |
| 原来就有五件套的行为 | 同步该特性 `spec.md` / `acceptance-tests.md` |
| 探查笔记 | 可先 `docs/research/`；写进正式约定后必须回写 design |
| 只重构、对外行为不变 | 可无行为 diff，但须写明「行为不变」与验证方式；禁止借重构改行为 |

## 2. 先分清两件事（别混）

| 你在做的事 | 人话 | 怎么才算做完 |
|------------|------|----------------|
| **改一个需求 / 修一个缺陷**（日常） | 这一票怎么安全改完？ | 有审过的「现状→目标态」；代码、测试、相关文档同一批合并 |
| **把老仓库逐步改成能稳用 AI**（整仓） | 这个仓以后怎么持续、安全地改？ | 模块列清、推进方式选好、主流程有书面现状、文档与真实行为对得上（§3.5） |

日常走 **改一个需求**（§5 起）。  
排「历史项目 AI Coding 化」走 **整仓推进**（§3），再拆成很多次「改一个需求」。

```text
【整仓】列清单 → 贴标签 → 选 S1–S5 → 写清主流程现状 → 拆小批
              │
              ▼ 每一批变成多次
【改一单】用 P1–P6 查现状 → 写目标态 → 让 AI 挑毛病 → 改代码 → 文档同批合并
```

## 3. 整仓改造怎么推进（技术经理）

目标：不是一天重写完，而是以后每次改都能按纪律做，改完后文档和真实行为对得上。

### 3.1 按顺序 5 步

| 步 | 做什么（人话） | 产出 | 谁拍板 |
|----|----------------|------|--------|
| **1. 列清单** | 有哪些模块、主流程、对外接口；旧文档还信不信 | 清单（可先粗） | 研发列，技术经理过一眼 |
| **2. 贴标签** | 每块只贴下面五类之一 | 标签 | 技术经理 |
| **3. 选推进方式** | 从 S1–S5 里选（§3.2） | 批次计划 | 技术经理 |
| **4. 写清主流程现状** | 用 §4 的办法；主流程写不清不准开大改 | 现状说明 / research | 研发 |
| **5. 拆小批做完** | 每批 = 若干次「改一个需求」；合入即改 design | MR 批次 | 研发执行 |

**标签（每块只贴一个）：**

| 标签 | 含义 | 默认怎么走 |
|------|------|------------|
| 要改行为 | 用户/调用方可见行为要变 | 现状→目标态，按单闭环 |
| 修缺陷 | 修错：先写清「错的现状」，目标态=正确行为 | 同上 |
| 重构但对外不变 | 对内理顺，对外不变 | 写明「行为不变」+ 测试锁住；禁止借机改行为 |
| 其实是新功能 | 等价新模块 | **走新项目流程（S5）** |
| 不再维护 | 退役 / 只读 | 标明不改；别让 AI 「顺手清理」 |

### 3.2 推进方式一共 5 种（S1–S5）

| # | 人话 | 何时用 | 具体怎么做 | 风险 |
|---|------|--------|------------|------|
| **S1** | **来一个需求改一个** | 平时修修补补 | 每次严格走「改一个需求」 | 不强制改文档会堆文档债 |
| **S2** | **按模块一块块啃** | 系统改造某一业务域 | 列该域清单 → 写主流程现状 → 一批批改 | 切太大易变相新项目 |
| **S3** | **先把真实行为写进文档** | 代码乱但接口相对稳；或大改前 | 先按真实行为改 design/OpenAPI，再动代码 | 只写文档不核实会美化现状 |
| **S4** | **先用测试锁住再动内部** | 重构 / 换实现 / 升依赖 | 测试锁现状 → 重构 → 保持全绿 | 测不到的边角仍可能漂 |
| **S5** | **按新项目流程做** | 新模块 / 现状写不清 / 跨域里程碑 | 停用本模板，走 [新项目](./new-project.md) | 硬套历史模板会假完成 |

**怎么选：**

- 平时交付：**S1**，且每张 MR 强制回写 design。  
- 某业务域要系统化搞好：**S2 + S3**。  
- 大重构、对外先不变：**S4**；行为也要变则拆成「先锁旧 → 再单独改行为」。  
- 新功能：**只走 S5**。

### 3.3 一批怎么切

1. 先主流程、后边角——主流程现状写不清，不准开大改。  
2. 先接口/权限/数据，后纯 UI 文案。  
3. 一批一个能演示的目标（如「列表筛选与接口约定对齐」），不要「本周重构账号域」。  
4. 每批结束能勾：验收通过 + 相关 design/spec 已更新或写明「无契约变更」。  
5. 连续两批现状写不动 → 停，改调研或 S5，不要硬写目标态。

### 3.4 文档债怎么还

| 优先级 | 做什么 | 说明 |
|--------|--------|------|
| P0 | 本次碰到的接口 / 表 / 权限 | 与代码同批，无例外 |
| P1 | 主流程已核实但 design 仍错/空 | 可开「只改文档」MR（S3） |
| P2 | 边角、罕见分支 | 记 research 或开放问题，挂主责 |

**禁止：**「代码先合，文档下个迭代补」——历史仓里下个迭代通常不会来。

### 3.5 某个模块何时算「改造阶段完成」

不是「用过 AI 了」，而是同时满足：

- [ ] 清单里该模块已贴标签，没有「说不清」悬空（或已挂主责）  
- [ ] 主流程有可引用的现状说明（文档或测试），并写明用了 P1–P6 哪几种  
- [ ] 后来的变更能出示审过的「现状→目标态」（或「行为不变」声明）  
- [ ] 对外行为与 `docs/design`（及 OpenAPI）无已知对不上；有则已进 P0/P1  
- [ ] 团队执行：没有上述材料就不让 AI 大改  

整仓可以一直「进行中」；**按模块勾完成**即可。

## 4. 现状怎么查清：一共 6 种办法（P1–P6）

写现状前先选办法。**总共 6 种**；可组合，模板里必须写明主要靠哪种、辅助靠哪种。

### 4.1 六种办法

| # | 人话 | 你具体做什么 | 最适合 | 能不能单独当真 |
|---|------|--------------|--------|----------------|
| **P1** | **自己跑一遍** | 点页面或调真实/预发接口，记输入→输出→边角 | 能上手的功能、缺陷现象 | 可以（默认首选） |
| **P2** | **测试 / 注释等旁证** | 有自动化测试就用；**没有也常见**——改看单测碎片、关键注释、TODO、示例请求/返回、fixture、断言文案，整理成「代码旁白里的约定」再核实。重构前若完全没有，可先补几条锁现状的单测 | 屎山无测时仍可用注释/示例；有测时重构前很有用 | 旁证可靠且已核实后可当辅证；**光靠注释不能定大改** |
| **P3** | **读代码推断** | 读实现与分支，写出「代码看起来会怎样」，标把握程度 | 难搭环境、批处理 | 不要单独定大改 |
| **P4** | **整理旧文档（可 AI 归纳）** | 旧 wiki/spec 常对不齐：把散落材料交给 AI **总结、去重、合并成一版候选现状**，再与真实行为对（一致/过期/冲突）。对得上的可回写成新版 design；对不上的标过期 | 文档多而乱、版本互相打架时 | **不能单独当真**（归纳稿仍是草稿，必须用 P1/P3 核实） |
| **P5** | **问现场的人** | 没有工单系统也没关系：问**现场运维、产品经理、熟手研发**——线上一直怎么用、哪些坑不能动；先写成待验证再核实 | 潜规则、无文档边角、只有人口头知道 | **不能单独当真** |
| **P6** | **看线上证据** | 日志、审计、trace、HAR、监控、错误码；写清时间段与条件 | 偶现、兼容旧客户端 | 偶现/仅生产时很有用 |

**硬规则：**

1. 能点页面 / 调接口 → 优先 P1；要可重复回归再补 P2（无测则先整理注释/示例，或补锁现状的单测）。  
2. P4、P5 只能当线索，不能当唯一真相；P4 的 AI 归纳稿也一样。  
3. P3 与 P1 冲突 → 以能复现的 P1/P6 为准，并写「文档/注释过期」。  
4. P6 与本地不一致 → 写明环境差，改之前先定以哪套为准。  
5. 模板必须填：**证据 = 主 P? + 辅 P?**。  
6. 没有自动化测试、没有工单——不妨碍走 P2（注释等旁证）和 P5（问运维/产品）。

### 4.2 怎么选

| 你现在的情况 | 主要用 | 再补 |
|--------------|--------|------|
| 本地/预发能点出来 | **P1** | P4：旧文档乱可先让 AI 归纳再对 |
| 有靠谱测试，或至少有关键注释/示例 | **P2** | P1 抽查主流程；要重构尽量先补锁现状单测 |
| 环境搭不起，只能读代码 | **P3** | P5 问运维/产品；有条件补 P6 |
| 旧文档又多又对不齐 | **P4 先 AI 归纳成候选**，再加 **P1 或 P3 核实** | 核实后可回写成新版 design |
| 「运维/产品说以前就这样」 | **P5 → 再用 P1/P3 核实** | — |
| 偶现 / 只有生产有 | **P6** | 能复现再补 P1 |
| 多端/旧客户端 | **P1 + P6** | P4 |

### 4.3 常用组合（不必发明第 7 种）

| 场景 | 组合 |
|------|------|
| 绝大多数改行为 | P1（主）+ P4（旧文档可先 AI 归纳再对） |
| 重构保行为（有测或先补测） | P2（主）+ P1（抽查） |
| 没人熟、文档假 | P5（运维/产品）→ P3 → P1 或 P6 |
| design 与线上不一致 | P1/P6（事实）对照 P4（归纳后的旧文档）→ 差额进 research 或走 S3 |

探查笔记进 `docs/research/` 只是存放位置，取证办法仍只有 P1–P6。

### 4.4 现状写到什么算合格

- [ ] 写的是**今天能观察到的行为**，不是实现八卦  
- [ ] 标明证据：`P1已复现` / `P2用例名` / `P3文件+结论` / `P6查询条件` 等  
- [ ] 主流程 + 已知边角（空数据、权限、超时、旧数据）  
- [ ] 「已知不合理但本期先不修」已列出（防 AI 顺手修好）  
- [ ] 与旧文档：一致 / 文档过期（说明）  
- [ ] 没把握时：缩小目标态，或先调研 / 走新项目——**不拿猜测当现状**  

### 4.5 协助查现状的提示词（可选）

```text
我在为遗留变更写现状（as-is）。请按取证办法帮我整理，不要编造未观察到的行为。
已知：【粘贴现象 / 链接 / 文件路径 / 日志摘要】
请输出：
1) 建议主要办法与辅助办法（仅从 P1 自己跑一遍 / P2 测试·注释等旁证 / P3 读代码 /
   P4 整理旧文档可 AI 归纳 / P5 问运维或产品 / P6 看线上证据 中选）；
2) 一份可写入模板的现状草稿（主流程 + 边角 + 本期先不修的坑）；
3) 仍未知、必须用人复现或问运维/产品的清单；
4) 禁止假设线上行为；不知道就标【未知】。
```

## 5. 日常：改一个需求的 6 步

```text
① 用 P1–P6 查清现状
        │
        ▼
② 写目标态（含影响面、本次不做、验收）
        │
        ▼
③★ 让模型只提问 → 补洞 → 人审过再开工
        │
        ▼
④ 列出要改的 design / spec
        │
        ▼
⑤ 提示词挂文档 → 按范围改代码
        │
        ▼
⑥ 对照验收；MR 文档与代码同批
```

| 步 | 主责 | 关键产出 | AI |
|----|------|----------|-----|
| ① 查现状 | 研发（产品可确认现象） | 写明用了哪几种 P 的现状 | 可协助整理，人确认现象 |
| ② 填模板 | 研发 / 产品 | 现状→目标态初稿 | 可起草，人改 |
| ③★ 挑毛病 | 研发 + 产品 | 未知关闭或挂起；文档冻结 | **必须反向追问** |
| ④ 文档清单 | 研发 | 要改的 design/spec 列表 | 可列建议 |
| ⑤ 实现 | 研发 + AI | 代码与测试 | 按文档改；越界就停 |
| ⑥ 验收 | 研发（产品可抽验） | 验收勾选；MR | 可跑测，人判定 |

## 6. 写清目标态与边界

目标态尽量与现状 **一条对一条**。

必须写清：

- **变更点**：相对现状，到底哪几条变了  
- **本次不做（非目标）**：AI 最爱顺手做的事列进禁止  
- **影响面**：页面 / 接口 / 数据 / 权限 / 下游 / 文档 —— 每一行「动/不动 + 说明」  
- **验收**：每条可打勾、也可判定失败  

不写「本次不做」= 默许 AI 随便加戏。

## 7. 阶段 ③★ — 大模型反向提问（关键关卡）

模板填完后，**先不要改代码**。让模型基于 as-is→to-be 做反向提问，逼出遗漏的边角、兼容与影响面。

### 7.1 推荐提示词

```text
你是苛刻的遗留系统变更审稿人。请只基于我粘贴的 as-is → to-be 文档，列出反向问题。要求：
1) 覆盖：as-is 证据路径是否足够（P1–P6）、是否可复现、to-be 是否可观察、一对一 diff 是否清晰、
   非目标是否足以挡住顺手重构、影响面是否漏了契约/数据/下游/权限、
   兼容旧数据与并发、回滚方式、验收是否可失败；
2) 每个问题说明：不问清楚时 AI 最可能臆造或误改什么；
3) 区分：必须本期拍板 / 可挂起；
4) 禁止直接改代码或扩大 to-be 范围；未知就提问。
输出：问题清单 + 建议回写到模板哪一节。
```

### 7.2 通过标准

- [ ] 至少一轮有记录的反向提问与答复  
- [ ] 阻塞实现的未知已关闭或显式挂起并指定主责  
- [ ] 人确认：可以挂上该文档让 AI 改代码

## 8. 阶段 ④ — design / spec / research 同步

历史变更同样遵守文档权威（详见新项目篇 §1）：

| 变更类型 | 必须同步 |
|----------|----------|
| HTTP 行为 / 字段 / 错误码 | `design/api*` + OpenAPI |
| 表结构 | `data-models`（及 migration）；特性 extensions 若有则更新 |
| 用户可见行为且已有五件套 | 该特性 `spec` / `acceptance-tests` |
| 架构级取舍 | ADR |
| 行为探查过程材料 | 可进 `research/`，采纳后回写 design |

**禁止：** 代码已合、契约文档仍描述旧行为。

## 9. 阶段 ⑤–⑥ — 实现与验收

### 9.1 实现节奏

```text
提示词严格挂 as-is→to-be（+ 列出的 design 路径）
  → 能写回归测则先红（失败用例证明旧行为或缺口）
  → 最小 diff 实现 to-be（绿）
  → 禁止范围外重构
  → 对照验收勾选
  → MR：代码 + 文档同批
```

### 9.2 给 AI 的约束（可粘贴）

```text
请严格依据本 as-is → to-be 文档实现变更。
只修改「影响面」标明的范围；「非目标」禁止实现。
不要引入文档未描述的字段、接口、菜单或行为。
若 as-is 与代码不符，先停下来说明差异，不要猜测后继续大改。
若需改契约或表结构，先说明并列出要同步的 docs/design（或 OpenAPI）文件，再改代码。
完成后对照「验收对照」逐条说明如何验证。
```

### 9.3 工具（与新项目同一纪律，场景不同）

| 技能 | 用在历史变更时 |
|------|----------------|
| **Superpowers** `systematic-debugging` | 复现、取证、定位根因——先证据后改 |
| `test-driven-development` | 回归：先锁行为再改 |
| `verification-before-completion` | 宣称完成前跑验证、给证据 |
| `brainstorming` | 方案有多条改法、影响面不清时先澄清 |
| **Frontend Design** | 仅当本次 to-be **明确包含**显著 UI 变更；且服从现有 Shell/设计系统，禁止借机整站换皮 |

不另开「其他推荐」清单；仓库 `AGENTS.md` 与 design 权威顺序始终生效。

## 11. 角色一览

| 角色 | 负责 | 默认不负责 |
|------|------|------------|
| 产品 | 确认现象与验收口径；答复业务向反向提问 | 代替研发读遍代码细节 |
| 技术经理 | 仓库级分桶与 S1–S5 策略；判定是否升级新项目；契约争议拍板 | 每个小 bug 都填模板（可授权研发自填） |
| 研发 | 选 P1–P6 取证、填模板、反向提问回写、实现、文档同步 | 无 as-is→to-be 就让 AI 大改 |
| AI | 协助读代码、整理取证、反向提问、按文档最小 diff | 当现状真相源、扩大非目标 |

## 12. 反模式（别这么干）

- 「就改一小下」→ 无文档直接让 AI 扫仓库改  
- 现状只抄旧文档或只信 AI 归纳稿（裸 P4）、不核实  
- 把运维/产品口头说法（P5）当冻结真相，不验证  
- 因为「没有自动化测试 / 没有工单」就空着 P2、P5 不用  
- to-be 只写「优化体验 / 更合理」，无可观察标准  
- 不写非目标，默认允许重构与加戏  
- 改了 API/表却不回写 design / OpenAPI  
- 把历史大改硬塞进本模板，逃避 PM 输入与五件套  
- 宣称「整仓改造完成」但主路径无 as-is 基线、design 仍与行为分叉  
- 用 Frontend Design 「顺便重做整个后台视觉」

## 13. 与新项目流程的边界

| | 新项目流程 | 历史项目维护 |
|--|-------------|---------------|
| 起点 | 产品背景 + 技术确认 | 取证写 as-is（P1–P6） |
| 核心文档 | PM 输入 → 五件套 | as-is → to-be |
| 关键关卡 | 对 PM 输入反向提问 | 对 as-is→to-be 反向提问 |
| 实现 | 按五件套 TDD | 按 to-be + 回归 / 验收 |
| 仓库级 | 能力从 003+ 登记推进 | 盘点分桶 + S1–S5 策略分批 |
| 禁止 | 无冻结 PM 输入就开写 | 无冻结 as-is→to-be 就大改 |
| 升级条件 | — | 等价新模块 → 走新项目（S5） |

铁律同 [为什么做与思维转变](./mindset.md)：没写清的行为，不让 AI 大改。
