## 文档权威顺序

<!-- 冲突时以此为准。agent 必须按此顺序判断哪份文档是真相 -->

```text
本特性 spec.md + acceptance-tests.md
        ↓
docs/design/* + OpenAPI
        ↓
系统级约定（Shell / 鉴权等）
        ↓
编码 skills
```

- 特性五件套**引用** design，不复制全局字段
- 破坏性变更必须先改 design / OpenAPI，再改代码
- `docs/research/` 是非正式基线；采纳为架构或 API 行为时必须回写 `docs/design/`

## 三条铁律（全员）

1. **没有写清的行为，不让 AI 大改**
   - 新项目：没有产品审定的 `pm-requirements-input.md`（且经过反向提问补洞），不开写五件套
   - 历史项目：没有研发审定的 as-is→to-be（含影响面、非目标、验收），不让 AI 大改老代码
2. **契约 / 验收变了，文档与代码同批更新**
   - HTTP 行为 / 字段 / 错误码 → `docs/design/api-contracts.md` + OpenAPI
   - 表结构 → `docs/design/data-models.md`（及 migration）
   - 用户可见行为且已有五件套 → 该特性 `spec.md` / `acceptance-tests.md`
   - 架构级取舍 → ADR（`docs/design/decisions/`）
3. **禁止臆造**：文档里没有的字段、接口、菜单、权限码，不要顺手加

## 禁止事项（全员红线）

<!-- 九条红线速查。任何一条触发 = 这单不算交付。新成员第一周必读 -->

- 臆造文档中不存在的字段 / 菜单 / 权限码
- 契约已变却只改代码
- 口头改需求不回写文档
- 跳过验收「先合并再说」
- 无「现状 → 目标态」就对遗留模块大范围自动重构
- 无审定 PM 输入就开写新特性五件套
- 因为「没有自动化测试 / 没有工单」就空着 P2、P5 不用
- 把 AI 归纳的旧文档稿或人口头说法直接当冻结真相
- 借 Frontend Design「顺便重做整站视觉」（除非目标态白纸黑字写了）

> 与「三条铁律」的关系：三条铁律是**正向纪律**，九条红线是**反向速查**——遇到具体动作时直接对着勾。

### 新项目主路径

```text
背景 → 技术确认 → assets → PM 输入草稿
  → ★ 反向提问并冻结 PM 输入
  → 五件套 + 同步 design（按需 research）
  → 按五件套实现（能测则先红后绿）→ 验收同批
```

五件套：`spec.md` · `plan.md` · `tasks.md` · `data-model-extensions.md` · `acceptance-tests.md`

### 历史：改一个需求 6 步

```text
查清现状 → 写目标态 → ★ 让 AI 挑毛病并冻结
  → 列出要改的文档 → 按文档改代码 → 验收并和文档一起合并
```

## Agent 工作准则

| 准则         | 一句话                              |
| ------------ | ----------------------------------- |
| 落盘优先     | 结论写文件，不靠聊天记忆            |
| 防发散       | 偏离计划时先判断再行动              |
| 基于原始代码 | 重构调研以代码为 ground truth       |
| 控制上下文   | 文档膨胀时及时拆分归档              |
| 文档持续重构 | 随执行深入修正和重组文档            |
| 深度对齐     | 一次问一个问题，提供推荐答案        |
| 测试先行     | 先写失败测试，再写实现              |
| 系统化调试   | 复现→最小化→假设→插桩→修复→回归测试 |
| 禁止臆造     | 文档没有的字段/接口/菜单不要加      |
| 文档同批     | 契约变了，文档与代码一起合并        |

## 任务管理过程

<!-- 把 plan.md 的里程碑切片落到可跟踪的执行节奏。文档是真相源，不在聊天里追进度 -->
<!-- 注：本段是 skill 新增的辅助段，非培训材料原话。不需要可整段删 -->

### 切片到执行的映射

| 层级     | 文件                     | 作用                                                         | 维护人                 |
| -------- | ------------------------ | ------------------------------------------------------------ | ---------------------- |
| 里程碑   | `plan.md` 的 M1 / M2…    | 切出可独立验收的交付批次，每批有"完成什么样 + 验收"          | 研发（技术经理审定）   |
| 任务     | `tasks.md` 的 T1 / T2…   | 里程碑内可勾选的最小单元，分组（后端/前端/测试/文档/DevOps） | 研发                   |
| 当前焦点 | 本文件 `当前任务焦点` 段 | 正在做的那个任务 + 下一步——让 agent 不重读全部文档就能定位   | 研发（每次会话前更新） |

### 执行节奏

1. 开工前：`plan.md` 切好里程碑，`tasks.md` 列出第一批任务
2. 每次会话开始：更新 `当前任务焦点`，指明正在做的任务对应的 `tasks.md#T?`
3. 每完成一个任务：在 `tasks.md` 勾选 `[x]`，同步更新 `当前任务焦点`
4. 每完成一个里程碑：对照 `plan.md` 的验收条目，确认后再开下一切片
5. 遇到计划外发现：先回写 `plan.md`（新增风险/前置/切片调整），再继续

### 历史项目的简化

历史项目走 as-is→to-be，不一定有完整 `plan.md`。此时：

- 用 `legacy-change.md` 的「验收对照」表当里程碑
- 每完成一条验收打勾，文档与代码同批合并

### 默认推荐（按主路径区分）

| 主路径         | 分支策略                                          | 理由                                                         |
| -------------- | ------------------------------------------------- | ------------------------------------------------------------ |
| **新项目**     | `feature/NNN-短名` 长分支，里程碑切片在分支内推进 | 五件套冻结后范围稳定，一个特性一个分支，MR 一次合并文档+代码 |
| **历史项目**   | `fix/短名` 或 `refactor/短名` 短分支              | as-is→to-be 范围小，快进快出，避免长分支与主线分叉太久       |
| **实验性重构** | `git worktree` 隔离                               | 不污染主分支；验证后决定是否合入                             |

### 通用规则（两条主路径都适用）

1. **文档与代码同批**：MR 必须同时包含文档变更（spec / design / OpenAPI），否则不算完成
2. **MR 描述挂文档链接**：MR 描述里必须链接到对应的 `spec.md` 或 `legacy-change.md`，方便审查时定位真相源
