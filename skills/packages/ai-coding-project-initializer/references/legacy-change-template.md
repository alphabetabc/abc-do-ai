# 历史变更使用指南

> 历史变更模板在 [templates/legacy-change.md](./templates/legacy-change.md)，由 `legacy-change.mjs` 复制生成。
> 本文件只放模板里没有的独家内容：P1–P6 / S1–S5 速查、反向提问提示词、design 同步规则、反模式。

## 三条硬规矩

1. 没有写清并审过的 **现状 → 目标态**（含影响面、非目标、验收），不让 AI 大改老代码。
2. 行为 / 接口约定已变却不回写 `docs/design` 或相关 spec，这单不算做完。
3. 其实是一整块新功能时，**升级走新项目流程**，不要硬套本模板。

## 何时用本模板 / 何时改走新项目

### 用本模板

- 改旧行为、缺陷里的行为变更
- 规格散落在代码与口头中，短期写不齐完整五件套
- 影响面圈得住、验收能写成对照表的增量改造

### 改走新项目

出现任一情况，走新项目主路径（PM 输入 → 五件套）：

- 新增一整块用户能感知的能力（不只是改旧行为）
- 要新的主数据模型 / 一批新 API，而非局部字段
- 跨很多业务域、需要独立里程碑
- 现状本身写不清——应先调研 / 补规格，不要硬写目标态

## 现状怎么查清：6 种办法（P1–P6）

| # | 人话 | 你具体做什么 | 能不能单独当真 |
|---|------|--------------|----------------|
| **P1** | **自己跑一遍** | 点页面或调真实/预发接口，记输入→输出→边角 | 可以（默认首选） |
| **P2** | **测试 / 注释等旁证** | 用自动化测试；没有则看单测碎片、关键注释、TODO、示例请求/返回、fixture | 核实后可当辅证；光靠注释不能定大改 |
| **P3** | **读代码推断** | 读实现与分支，写出"代码看起来会怎样"，标把握程度 | 不要单独定大改 |
| **P4** | **整理旧文档（可 AI 归纳）** | 把散落材料交给 AI 总结、去重、合并成一版候选现状，再与真实行为对 | **不能**（归纳稿必须用 P1/P3 核实） |
| **P5** | **问现场的人** | 问现场运维、产品经理、熟手研发；先写成待验证再核实 | **不能** |
| **P6** | **看线上证据** | 日志、审计、trace、HAR、监控、错误码 | 偶现时很有用 |

**硬规则：**

1. 能点页面 / 调接口 → 优先 P1
2. P4、P5 只能当线索，不能当唯一真相
3. P3 与 P1 冲突 → 以能复现的 P1/P6 为准，并写"文档/注释过期"
4. 模板必须填：**证据 = 主 P? + 辅 P?**
5. 没有自动化测试、没有工单——不妨碍走 P2 和 P5

## 整仓推进：5 种方式（S1–S5）

| # | 人话 | 何时用 |
|---|------|--------|
| **S1** | **来一个需求改一个** | 平时修修补补；每张 MR 强制回写 design |
| **S2** | **按模块一块块啃** | 系统改造某一业务域 |
| **S3** | **先把真实行为写进文档** | 代码乱但接口相对稳；或大改前 |
| **S4** | **先用测试锁住再动内部** | 重构 / 换实现 / 升依赖 |
| **S5** | **按新项目流程做** | 新模块 / 现状写不清 / 跨域里程碑 |

## 提示词（按阶段复制）

### 协助查清现状（选 P1–P6）

```text
我在为遗留变更写现状（as-is）。请按取证办法帮我整理，不要编造未观察到的行为。
已知：【粘贴现象 / 链接 / 文件路径 / 日志摘要 / 旧文档片段】
请输出：
1) 建议主要办法与辅助办法（仅从 P1 自己跑一遍 / P2 测试·注释等旁证 / P3 读代码 /
   P4 整理旧文档可 AI 归纳 / P5 问运维或产品 / P6 看线上证据 中选）；
2) 一份可写入模板的现状草稿（主流程 + 边角 + 本期先不修的坑）；
3) 仍未知、必须用人复现或问运维/产品的清单；
4) 禁止假设线上行为；不知道就标【未知】。
```

### 旧文档很乱 — 先归纳再核实（P4 辅助）

```text
下面是散落的旧文档 / wiki / 注释摘录（可能互相矛盾）。
请总结、去重，合并成一版「候选现状」草稿，并标明：
1) 各来源一致之处；
2) 互相冲突之处（列出冲突点，不要擅自挑边）；
3) 完全缺失、必须靠复现或问人才能补的缺口。
禁止把归纳结果说成已核实的线上真相；输出末尾加一句：须用 P1/P3 核实后再冻结。
```

### 现状→目标态写完后 — 反向提问（先于改代码）★

```text
你是苛刻的遗留系统变更审稿人。请只基于我粘贴的「现状 → 目标态」文档，列出反向问题。要求：
1) 覆盖：现状证据（P1–P6）是否足够、能否复现、目标态能否观察、一对一对照是否清晰、
   「本次不做」能否挡住顺手重构、影响面是否漏接口/数据/下游/权限、
   兼容与回滚、验收能否判定失败；
2) 每个问题说明：不问清楚时 AI 最可能误改什么；
3) 区分：必须本期拍板 / 可挂起；
4) 禁止直接改代码或扩大目标态范围；未知就提问。
输出：问题清单 + 建议回写到模板哪一节。
```

**通过标准：**

- [ ] 至少一轮有记录的反向提问与答复
- [ ] 阻塞实现的未知已关闭或显式挂起并指定主责
- [ ] 人确认：可以挂上该文档让 AI 改代码

## design / spec / research 同步

| 变更类型 | 必须同步 |
|----------|----------|
| HTTP 行为 / 字段 / 错误码 | `docs/design/api-contracts.md` + OpenAPI |
| 表结构 | `docs/design/data-models.md`（及 migration）；特性 extensions 若有则更新 |
| 用户可见行为且已有五件套 | 该特性 `spec.md` / `acceptance-tests.md` |
| 架构级取舍 | ADR（`docs/design/decisions/`） |
| 行为探查过程材料 | 可进 `docs/research/`，采纳后回写 design |

**禁止：** 代码已合、契约文档仍描述旧行为。

## 开工前打勾

- [ ] 已判定走历史路径（不是该升级新项目的 S5）
- [ ] 现状写明证据用了哪些 P；没把握未硬开大改
- [ ] 目标态 / 影响面 / 本次不做 / 验收已写清
- [ ] 已反向提问并回写；文档已冻结
- [ ] design / OpenAPI / spec 同步项已列出或已改
- [ ] 实现提示词挂了这份文档

## 反模式（别这么干）

- 「就改一小下」→ 无文档直接让 AI 扫仓库改
- 现状只抄旧文档或只信 AI 归纳稿（裸 P4）、不核实
- 把运维/产品口头说法（P5）当冻结真相，不验证
- 因为"没有自动化测试 / 没有工单"就空着 P2、P5 不用
- to-be 只写"优化体验 / 更合理"，无可观察标准
- 不写非目标，默认允许重构与加戏
- 改了 API/表却不回写 design / OpenAPI
- 把历史大改硬塞进本模板，逃避 PM 输入与五件套
- 宣称"整仓改造完成"但主路径无 as-is 基线、design 仍与行为分叉
- 借 Frontend Design"顺便重做整站视觉"（除非目标态白纸黑字写了）
