# 决策选项模板

> 用途：智能体给 PM 决策时，列出选项 + 推荐 + 理由 + 等待 PM 选择

---

## 模板

```
[decision] 需要你拍板：<决策标题>

选项：
- A: <选项 A 描述>
- B: <选项 B 描述>
- C: <选项 C 描述>（如有）

推荐：A
理由：<为什么推荐 A>

影响范围：
- 选 A：<影响 1>、<影响 2>
- 选 B：<影响 1>、<影响 2>
- 选 C：<影响 1>、<影响 2>

请选择：[A/B/C]
```

---

## 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 决策标题 | 是 | 一句话说清要决策什么 |
| 选项 | 是 | 至少 2 个，最多 4 个（PM 决策疲劳）|
| 推荐 | 是 | 智能体必须给推荐（不替 PM 拍板，但给建议）|
| 理由 | 是 | 为什么推荐，1~3 句话 |
| 影响范围 | 是 | 每个选项的影响（PM 决定时考虑）|
| 等待 PM 选择 | 是 | 必须显式等待，不替 PM 选 |

---

## 填写示例

### 示例 1：技术选型

```
[decision] 需要你拍板：前端框架选型

选项：
- A: React 18 + Vite + TypeScript（生态最丰富，团队学习成本中）
- B: Vue 3 + Vite + TypeScript（上手快，团队学习成本低）
- C: Vanilla JS + Vite（最轻量，但 UI 库需自己选）

推荐：B
理由：团队有 Vue 经验 + 项目复杂度中等 + 启动快 + 中文文档丰富

影响范围：
- 选 A：招聘 React 工程师 1~2 周、培养 1~2 周、生态最丰富
- 选 B：现有 Vue 工程师可直接上手、文档最全、招聘容易
- 选 C：最轻量但需自选 UI 库，长期维护成本高

请选择：[A/B/C]
```

### 示例 2：模块拆分

```
[decision] 需要你拍板：客户运营模块是否拆为两个子模块

选项：
- A: 拆为"客户管理" + "运营活动"两个子模块（边界清晰）
- B: 保持"客户运营"一个模块（简单）

推荐：A
理由：未来客户和运营是两个独立团队负责，预先拆分降低后期重构成本

影响范围：
- 选 A：路由结构更复杂、初始工作量 +20%、后期灵活
- 选 B：简单、初始工作量小、但未来重构需回滚

请选择：[A/B]
```

### 示例 3：API 设计

```
[decision] 需要你拍板：客户列表 API 返回结构

选项：
- A: 返回完整客户对象（{ id, name, phone, address, ... }）
- B: 返回精简对象（{ id, name }） + 详情走单独 API
- C: 走 GraphQL（按需返回）

推荐：B
理由：列表页不需要完整信息，减少带宽、详情页按需加载

影响范围：
- 选 A：列表加载慢、前端处理简单
- 选 B：列表快、需前端组合、API 多一个详情接口
- 选 C：最灵活但学习成本高

请选择：[A/B/C]
```

---

## 反模式

- ❌ 只给 1 个选项（不构成决策）
- ❌ 选项超过 4 个（PM 决策疲劳）
- ❌ 不给推荐（PM 没有方向）
- ❌ 不给理由（PM 不信智能体）
- ❌ 不列影响范围（PM 决策时信息不足）
- ❌ 替 PM 选（违反 §2.6 决策独立）
- ❌ 列出"非选项"（不应该有"建议不选"这种"伪选项"）
