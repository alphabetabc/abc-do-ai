# 04 · 执行 SOP 骨架

> 核心命题：从需求到交付，标准执行流程是什么

---

## 原则

**1. 闭环四阶段。**
- 扫描 → 规划 → 改造 → 验收
- 四阶段顺序执行，缺一不可
- 闭环：验收报告反馈到下一轮扫描的输入
怎么做：每个项目都按四阶段推进，不跳阶段。

**2. 三段式执行。**
- 前置：必读文档 + 文件盘点 + 基础设施文件先行
- 执行：按批次串行/并行
- 收尾：同步更新多份文档 + 记录 bug 案例 + 沉淀反模式
怎么做：每批次都有标准三段式，不漏段。

**3. 基础设施文件先行。**
- 入口文件、路由配置、API 客户端、模拟入口、命名空间 先建
- 没有这些，后续任务无法联调
怎么做：在执行第一个路由前先建好基础设施，并冒烟测试。

**4. 串行执行批次（除非显式并行）。**
- 默认批次间串行（依赖关系决定）
- 同批次内可并行（无依赖时）
- 显式声明才并行（避免意外冲突）
怎么做：批次计划里标注"串行 / 并行"，并行时声明互斥资源。

**5. 遇问题建 issue 不卡住。**
- 遇到问题立即建 issue 跟踪
- 不停下来等决策（除非命中升级白名单）
- 临时方案 + issue 标记 + 继续推进
怎么做：维护 issues 目录 + 标准模板，issue 不阻塞后续任务。

**6. 决策回溯。**
- 每个决策都记录：场景 + 选项 + 选定 + 理由
- 决策日志可追溯
- 推翻旧决策必须留变更说明
怎么做：决策日志是"决策的单一真相源"。

**7. 自更新触发点（每个执行者的标准动作）。**
- 任务完成 → 更新进度日志
- 产生新决策 → 写入决策日志
- 遇到 bug → 记录 bug 案例
- 发现反模式 → 抽象反模式条目
- 完成扫描 → 输出扫描报告
怎么做：每个执行者都有自更新清单，作为交付的一部分。

**8. 验收 checklist。**
- 每个 Phase 都有硬性 checklist
- 验收前逐项核对
- 验收不通过不进入下一阶段
怎么做：checklist 模板 + 自动化核对（能自动化的部分）。

**9. 错误回滚机制。**
- 批次级 / 路由级 / 字段级 checkpoint
- 改坏时一键回滚到上一个稳定状态
- 危险动作（删除、覆盖、命名推翻）二次确认
怎么做：git + checkpoint 文件 + 危险动作黑名单。

**10. 交付即锁定。**
- Phase 交付时打基线（git tag + 锁定文档）
- 锁定后只允许 bug 修复
- 大动作变更必须新建 Phase（不破坏已锁定的基线）
怎么做：git tag + 锁定文档 + 强校验。

**11. 跨任务交接。**
- 同任务交接：切换包（见 01 §3）
- 不同任务交接：交付报告 + 知识库更新
- 跨 PM 交接：完整工作区快照
  怎么做：每种交接类型都有标准包结构。

**12. 动手三件套（build-agent 等执行智能体的标准动作）。**
- **① PM 授权检查** —— 验证"改这个文件"在 PM 授权范围内
- **② 上下文对齐检查** —— 验证"改的内容" vs PM 当前聊天上下文
- **③ 报告准备** —— 准备好"改了哪些 + 为什么改 + 影响范围"模板
- **绝对禁止**跳步：跳过 ① = 擅自改文件；跳过 ② = 文不对题；跳过 ③ = 改完不报告
  怎么做：build-agent 改任何文件前**必做**这三步。详见 [08-pm-authorization.md](./08-pm-authorization.md) + [09-context-alignment.md](./09-context-alignment.md)。
