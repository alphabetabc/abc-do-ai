# 01 · 角色与边界

> 核心命题：谁提需求 / 谁执行 / 谁决策 / 何时升级

---

## 原则

**1. 角色三层分离。**

- 需求方（PM/业务）：提需求、决策、验收
- 执行方（AI/工程师）：执行任务、汇报进度、回填产物
- 决策方（通常 = 需求方，特殊情况升级）：拍板架构级变更与白名单事项
  怎么做：每份任务包都明确标注"角色 + 边界"，不留灰色地带。

**2. 统一入口指令。**

- 所有执行者从同一份入口指令文件开始
- 入口指令包含：任务范围、必读文档、读取顺序、退出标准
- 接手者读完即可开工，无需现场问
  怎么做：把入口指令放在固定路径（如 `<项目配置目录>/<project>/entry.md`，具体路径由所用工具决定）。

**3. 切换包机制。**

- 执行者上下文窗口快满时，生成切换包
- 切换包包含：状态 + 待办 + 决策记录 + 风险点 + 下一步
- 新执行者读切换包即可接手
  怎么做：定义切换包标准结构（见 04 §9 / handover 模板）。

**4. 自主决策 vs 升级白名单。**

- 默认自主：命名、端点拆分、数据分类、文档更新、字段调整
- 必须升级：架构变更、模块拆分、命名推翻、决策冲突
  怎么做：维护一份"自主 vs 升级"对照表（见 07）。

**5. 多人协作隔离。**

- 每个 PM / 每个任务分配独立工作区
- 工作区之间不共享状态
- 跨工作区协作通过知识库或交接包
  怎么做：用工作区 ID / 命名空间做隔离层。

**6. PM 体验原则。**

- PM 只跟一个智能体 / 入口对话
- 子执行者在后台自己串起来
- PM 不感知底层的拆分与并行
  怎么做：调度器（Orchestrator）统一对外，子执行者不直接对 PM 暴露。

**7. 智能体主动权可用（可主动打断 PM）。**

- 智能体**有主动权**：可以主动反馈、主动报告、主动建议、主动警告
- 但**必须有意义**（不能为了刷存在感而打断）
- 打断时**必须说明原因**
- 同一任务最多每 5 分钟打断 1 次（防止刷屏）
  怎么做：每次打断输出"打断标签 + 打断原因 + 后续选项"。详见 [10-proactive-interrupt.md](./10-proactive-interrupt.md)。

**8. 角色可继承不可越权。**

- 下游角色可继承上游角色的部分权限
- 下游不可推翻上游已锁定的决策
- 越权时必须升级
  怎么做：定义角色权限矩阵 + 越权触发升级。
