# 05 · 文档体系设计

> 核心命题：多份文档怎么分工、怎么保持一致、怎么可追溯

---

## 原则

**1. 七件套分工。**
- roadmap（总览）/ detailed-plan（细粒度）/ status（状态）/ scan-report（扫描）/ runtime-bug（bug 库）/ issues（问题跟踪）/ phase-mapping（编号对照）
- 每件套职责单一、互不重复
- 任何项目都从这 7 类里挑需要用到的，不全用也不强求新增
怎么做：定义每件套的输入、输出、维护责任人。

**2. canonical 编号 + 快捷查询。**
- 编号有歧义时，以"权威表"（canonical 表）为准
- 编号错位时建对照表纠正（不强行改历史文件）
- 提供"快捷查询"段加速接手者定位
怎么做：维护 canonical 编号表 + 快捷查询段。

**3. 多文档数字一致性。**
- 同一数字在多文档出现时必须一致
- 任何修改同步所有引用
- 数字变更触发交叉检查
怎么做：每次更新后做"数字一致性"核对清单。

**4. 决策日志独立成档。**
- 所有决策都进决策日志
- 决策日志按时间倒序
- 决策可被推翻但必须留痕（旧决策 + 推翻说明 + 新决策）
怎么做：决策日志是"决策的单一真相源"，不允许口头决策。

**5. issue 跟踪独立目录。**
- 遇到的所有问题建 issue 文件
- issue 包含：现象 + 临时处理 + 待 review 选项 + 回填指引
- issue 不阻塞后续任务
怎么做：issues 目录 + 标准模板 + 不允许口头遗留问题。

**6. 扫描报告 vs 详细计划分离。**
- 扫描报告：发现阶段（数据迁移分类、风险点、字段表）
- 详细计划：执行阶段（端点数、批次划分、命名规范）
- 两份不混合、不重复
怎么做：每个 Phase 都有对应两份文档，扫描在前、计划在后。

**7. 进度速查表（单一真相源）。**
- 一张表看全项目进度
- 字段：模块 / 路由数 / 已 Mock / 未 Mock / Mock 率
- 每次更新都重算合计
怎么做：固定一张表，所有变更同步更新，禁止散落多表。

**8. 锁定基线文档。**
- 跨阶段时生成基线文档
- 基线包含：架构决策、接口契约、模块边界、不可变更项
- 基线与 git tag 一一对应
怎么做：基线模板 + git tag 自动化 + 强校验。

**9. 文档版本化。**
- 所有文档都进版本控制
- 关键决策打 tag
- 文档变更可追溯
怎么做：git + 强制 review + 变更说明。

**10. 文档维护责任人。**
- 每份文档明确责任人
- 责任人负责一致性、版本、归档
- 责任人变更时交接
怎么做：文档头标注"维护人 + 维护周期 + 最近一次 review"。

**11. 文档的反模式。**
- ❌ 一份文档里既讲"是什么"又讲"怎么做"——拆成两份
- ❌ 多份文档讲同一件事——只保留最权威的一份
- ❌ 文档无更新日期——强制加时间戳
- ❌ 决策只口头不下文——强制落档
怎么做：定期 review 文档，识别并消除反模式。
