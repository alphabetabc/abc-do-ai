# 07 · 附录 · 决策白名单

> 核心命题：哪些事默认自主、哪些事必须升级到决策方

---

## 自主决策（默认执行，不询问）

| 场景 | 默认决策 |
|------|---------|
| 命名 | 沿用既有命名规范，冲突时优先沿用既有命名 |
| 端点拆分 | 按页面体量（参考 02 §4）|
| 数据迁移分类 | 按 03 §4 分类标准 |
| 文档更新 | 同步更新所有引用文档 |
| 单路由字段调整 | 仅影响本路由的字段 |
| 错误日志记录 | 按 06 标准记录 |
| 批次内并行 | 显式声明即可并行 |
| issue 跟踪 | 立即建 issue 不阻塞 |
| 自更新触发 | 按 04 §7 触发点 |
| 验收 checklist 核对 | 按既有 checklist 逐项核对 |

## 必须升级（命中即停下来询问决策方）

| 场景 | 为什么必须问 |
|------|-------------|
| 架构级变更（新增/删除/合并模块）| 影响 phase 编号和路线图 |
| 模块边界拆分/合并 | 影响 API/数据 文件结构 |
| 命名规范变更 | 影响所有路由 |
| 决策推翻既有结论 | 影响路线图与基线 |
| 端点数 > 5 或 < 1 | 超出默认范围 |
| 数据源无法定位 | 需决策方确认新增还是跳过 |
| 与自主决策规则冲突 | 任何冲突都需确认 |
| 锁定基线后做大动作变更 | 破坏已锁定的基线 |

## 升级流程

1. **命中白名单 → 立即建 issue 文件**（不要先停下来问）
2. **issue 包含**：现象 + 临时处理 + 待 review 选项 + 影响范围
3. **通过约定渠道**（会议 / 消息 / 工具）询问决策方
4. **决策落地后**回填到决策日志 + 相关文档
5. **禁止**直接修改文档猜测答案

## 决策落地后

- 回填到对应详细计划 / roadmap
- 必要时升级为新原则（沉淀到 01~06）
- 必要时升级本白名单（新增/删除某类场景）
- 同步给所有相关方

## 白名单的维护

- 定期 review：每 Phase 验收时 review 一次
- 触发升级：发现新类型场景时新增条目
- 触发降级：发现某类决策可标准化时移入"自主"区域
怎么做：白名单本身也是文档，进版本控制。

## 反模式（白名单的常见误用）

- ❌ "为了快"擅自做白名单事项 → 事后追责
- ❌ 频繁升级同一类问题 → 升级为新原则
- ❌ 升级但不建 issue → 决策无法追溯
- ❌ 决策后不回填 → 文档与实际脱节
怎么做：定期 review 白名单使用情况，识别并消除反模式。
