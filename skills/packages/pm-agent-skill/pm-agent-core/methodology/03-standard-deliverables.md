# 03 · 标准化产物

> 核心命题：每任务产出什么、怎么命名、数据 vs UI 怎么分

---

## 原则

**1. 每任务 4 件套产物。**
- 数据层（数据契约 / 数据文件）
- 接口层（API 端点 + 模拟数据）
- 视图层（页面文件 + 组件）
- 配置层（路由 / 菜单 / 权限 / 入口）
怎么做：4 件套缺一不可，每件套都有验收点。

**2. 命名规范四联动。**
- 端点 URL ↔ API 函数 ↔ 数据文件 ↔ 视图文件 命名必须对齐
- 一个改、四个都改
- 冲突时优先沿用既有命名（避免破坏下游引用）
怎么做：维护一份"命名映射表"，每次新增路由都登记。

**3. 数据语义 vs UI 状态分离。**
- 数据语义（业务数据、列表、统计、字典）→ 入 API
- UI 状态（表单输入、开关、选中项、图表实例）→ 留视图
- 静态常量（页面级选项、头部配置）→ 留视图
怎么做：每个状态变量都标注"数据"或"UI"。

**4. 数据迁移分类三档。**
- ✅ 入 API：业务数据、表格、统计、字典、趋势
- ❌ 留视图：表单、UI 开关、图表实例、选中项、分页交互
- ⚠️ 风险点：硬编码数组、状态/引用混用、组件引用、不可序列化字段
怎么做：每路由改造前先做分类，分类不通过不开始改造。

**5. 不可序列化字段不入 API。**
- icon / 组件引用 / 循环引用 / DOM 实例 / 函数
- 模拟数据不返回，视图端按 type/id 映射补回
怎么做：模拟数据序列化校验，失败即标记"不可返回"。

**6. 公共静态 vs 业务数据分别处理。**
- 公共静态（页面级选项、头部配置、纯展示文案）→ 留视图
- 业务数据（字典、列表、用户数据、跨页共享）→ 入 API
怎么做：以"是否依赖后端 / 是否跨页共享"为判断标准。

**7. 路由命名一致性。**
- URL 路径：kebab-case（短横线分隔）
- API 函数：camelCase + 业务前缀
- 数据文件：camelCase + 业务名
- 视图文件：PascalCase + 业务名
怎么做：定义命名转换表，全项目统一。

**8. 文件路径规范。**
- 业务页面：按业务域分子目录
- 公共组件：放统一 components 目录
- 工具函数：放统一 utils 目录
- 模拟数据 / API：按业务模块分子目录
怎么做：固定目录结构 + 禁止跨域引用。

**9. 命名冲突的优先级。**
- 同名冲突时优先沿用项目既有命名
- 与外部库同名时加业务前缀
- 与保留字冲突时换词而非缩写
怎么做：冲突时停下来升级，不擅自改命名。

**10. 产物的版本化。**
- 每件套产物都进版本控制
- 关键变更打 tag
- 产物变更可追溯
怎么做：git + 强制 review。

**11. 页面态标准（高保真原型硬指标）。**
- 每个有数据加载的页面必须覆盖 4 态：`loading` / `empty` / `error` / `success`
- 4 态定义：
  - **loading**：数据加载中（骨架屏 / spinner / 进度条）
  - **empty**：数据为空（空状态图 + 引导文案 + 主操作按钮）
  - **error**：加载或操作失败（错误提示 + 重试按钮）
  - **success**：数据正常（核心内容）
- 营销活动页等业务场景可扩展（如：`notStarted` / `ended` / `soldOut`）
- build-agent 输出 4 件套时，每个页面附"**页面态覆盖矩阵**"
- verify-agent checklist 必须包含"页面态 4 项齐全"
- 设计稿必须包含 4 态对应的视觉稿（如缺需 PM 补 / 智能体按通用模式生成）
怎么做：每个页面改造前先确认 4 态设计稿是否齐全；不齐的开 issue 跟踪，不阻塞当前任务。
