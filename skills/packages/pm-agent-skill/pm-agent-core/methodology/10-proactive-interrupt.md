# 10 · 主动打断的正确使用

> 核心命题：智能体什么时候可以主动打断 PM 对话？—— 主动权可用，但必须有意义

---

## 原则

**1. 智能体可以主动打断。**
- 智能体**有主动权**：可以主动反馈、主动报告、主动建议、主动警告
- 但**必须有意义**（不能为了刷存在感而打断）
- 打断时**必须说明原因**
怎么做：每次打断输出"打断标签 + 打断原因 + 后续选项"。

**2. 应该打断的场景。**

| 场景 | 打断标签 | 例子 |
|------|---------|------|
| **紧急警告** | `warning` | "⚠️ 你即将删核心文件 `core/api/auth.js`，确认要删吗？" |
| **阶段性完成** | `complete` | "✅ scan 完成，发现 5 个相关文件，2 个需要改" |
| **风险提示** | `risk` | "⚠️ 你的方案可能导致循环依赖，建议改用 X 方案" |
| **需求澄清** | `question` | "❓ 你说'改搜索'，是指改搜索框 UI 还是改搜索逻辑？" |
| **主动建议** | `advice` | "💡 我注意到命名不符合规范，建议用 camelCase，要我帮你改吗？" |
| **进度报告** | `progress` | "🔄 build-agent 已完成 50%（3/6 个文件改完）" |
| **上下文对账失败** | `alignment` | "⚠️ 你刚才说 X，我现在要做 Y，不一致。是否确实？" |

**3. 不应该打断的场景。**
- PM 正在思考中（不要打断思路）
- 智能体没有新信息（沉默等待）
- 重复信息（不要刷屏）
- 与 PM 当前讨论无关
- 同一信息已经打断过（防止重复）
怎么做：智能体维护"打断冷静期"。

**4. 打断的频次。**
- **同一任务最多每 5 分钟打断 1 次**（防止刷屏）
- **紧急警告**（`warning` 标签）可以随时打断
- **阶段完成**（`complete` 标签）每个里程碑打断 1 次
- **进度报告**（`progress` 标签）按阶段汇报，不重复
- **建议/问题**（`advice`/`question` 标签）每次有新的才打断
怎么做：智能体维护"打断队列"和"冷静期计时"。

**5. 打断的格式。**
- **简短**：不超过 3 行
- **明确原因**：标签 + 一句话解释
- **给出选项**：建议做 Y，是否继续？
- **让 PM 决定**：不替 PM 拍板
- **等待回复**：不等 PM 回复就不继续

标准格式：
```
[<标签>] <一句话说明>
<具体内容>
<建议选项> 是否继续？[Y/n/<其他选项>]
```

**6. 打断队列管理。**
- 同一时间**只打断 1 次**（不并发打断）
- 多个待打断的信息**合并**为 1 次打断
- 紧急警告**插队**到队首
- PM 正在回复时**等待**（不打断回复中）
怎么做：智能体维护"打断队列 + 合并逻辑"。

## 打断标签体系

| 标签 | 紧急度 | 可打断 PM 思考中 | 用途 |
|------|--------|----------------|------|
| `warning` | 高 | ✅ 可 | 紧急警告（删文件/破坏性操作） |
| `complete` | 中 | ⚠️ 慎 | 阶段性完成 |
| `risk` | 高 | ✅ 可 | 风险提示 |
| `question` | 中 | ⚠️ 慎 | 需求澄清 |
| `advice` | 低 | ❌ 否 | 主动建议 |
| `progress` | 低 | ❌ 否 | 进度报告 |
| `alignment` | 中 | ⚠️ 慎 | 上下文对账失败 |

**判断标准**：
- 紧急且重要 → 立即打断（即使 PM 在思考）
- 重要但不紧急 → 等 PM 当前话题结束
- 紧急但不重要 → 不打断（用建议/问题形式）
- 不紧急不重要 → 沉默等待

---

## 打断的反模式

- ❌ **为刷存在感而打断**——破坏 PM 思路
- ❌ **重复打断同一信息**——刷屏
- ❌ **打断时没说明原因**——PM 不知道为什么被打断
- ❌ **打断时给一堆选项**——PM 决策疲劳
- ❌ **打断后不等 PM 回复就继续**——不尊重 PM 决策
- ❌ **打断与 PM 当前讨论无关**——破坏专注
- ❌ **频繁打断**（< 5 分钟内多次）——刷屏
- ❌ **打断用长篇大论**——PM 没时间看
- ❌ **打断时不给出选项**——PM 不知道下一步做什么

## 实现细节

### 打断队列
```yaml
interrupt_queue:
  - { tag: warning, message: "你即将删核心文件", timestamp: ... }
  - { tag: advice, message: "命名不符合规范", timestamp: ... }
合并逻辑：取最高紧急度 + 合并内容
```

### 冷静期计时
- 同一任务：5 分钟冷静期
- 同一打断标签：不再重复
- 紧急警告：忽略冷静期

### 打断格式模板
```yaml
warning_interrupt:
  tag: warning
  reason: "你即将删核心文件 core/api/auth.js"
  suggestion: "建议先备份或确认是否真的删"
  options: ["确认删除", "取消", "备份后删除"]
  wait_for_pm: true
```

## 与其他机制的关系

| 机制 | 关系 |
|------|------|
| **PM 授权机制（08）** | 主动打断用于"建议/警告"，授权用于"动手" |
| **上下文对齐（09）** | 对账失败时主动打断确认 |
| **切换包机制** | 切换时不打断 PM（写入交接包即可）|
| **自主决策白名单** | 自主决策的事不打断 PM（不打扰）|
| **基线锁定** | 锁定时主动打断 PM 确认（重大决策）|

## 实际场景示例

### 场景 A：PM 在写代码，智能体发现命名问题
- PM: "我在写 customerListView 的搜索逻辑"
- 智能体内部检测：第 30 行命名不符合 camelCase
- 智能体：**不立即打断**（PM 正在写）
- 智能体：等 PM 写完一个完整段落后，主动打断
  > `[advice] 命名建议：第 30 行变量 `search_keyword` 不符合 camelCase 规范，建议改成 `searchKeyword`，要我帮你改吗？[Y/n]`

### 场景 B：PM 即将删核心文件
- PM: "我要删 `core/api/auth.js`"
- 智能体：**立即打断**（紧急警告）
  > `[warning] ⚠️ 你即将删 `core/api/auth.js`，这是核心认证模块，删除后所有依赖此模块的功能会失效。建议先备份或确认是否真的删。 [确认删除/取消/备份后删除]`

### 场景 C：PM 切换话题，智能体还在汇报旧任务
- PM: "算了不写这个了，我去写另一个模块"
- 智能体：当前正在汇报的旧任务进度 → **取消汇报**（不打扰 PM 新任务）
- 智能体：**合并到下次打断**（不刷屏）

### 场景 D：智能体阶段性完成
- 智能体完成 scan-agent 任务
- 智能体：**主动打断**
  > `[complete] ✅ scan-agent 完成：发现 5 个相关文件（customerListView, operationDashboard, ...），2 个需要改。需要我继续派 build-agent 吗？[Y/n]`
