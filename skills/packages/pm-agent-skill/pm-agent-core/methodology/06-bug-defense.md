# 06 · Bug 防御与知识沉淀

> 核心命题：怎么避免反复踩同一个坑、怎么把经验沉淀成可复用资产

---

## 原则

**1. 反模式清单。**
- 把踩过的坑抽象成"反模式"
- 每条反模式：场景 + 反例 + 正例 + 防复发
- 反模式可被新案例引用
怎么做：维护 anti-patterns 目录，每条反模式独立成档。

**2. 速查表（一线救命工具）。**
- 错误现象 → 根因 → 修复 一键查询
- 按错误类型分类（网络 / 数据 / UI / 性能 / 配置）
- 速查表优先级高于反模式（先救命、再抽象）
怎么做：维护 bug-cases 速查表，按错误现象关键词检索。

**3. 案例归档四字段。**
- 现象：具体报错或异常表现（含错误堆栈关键词）
- 根因：为什么会发生（追到具体技术原理）
- 修复：怎么改（具体到代码或配置）
- 防复发：怎么避免再犯（防御前置清单）
怎么做：每条案例 4 字段齐全，缺一不收档。

**4. 防御前置（改造前自查）。**
- 每个文件改造前先查反模式清单
- 防御 > 修复
- 改造后做防御自检
怎么做：改造 checklist 含"反模式自查"环节，强制执行。

**5. 知识库四分类。**
- bug-cases：具体案例（救命用）
- anti-patterns：抽象反模式（防御用）
- decisions：决策日志（追溯用）
- scanning：扫描报告（输入用）
- 4 类不混合、各自独立目录
怎么做：知识库分目录管理，目录即分类。

**6. 验收回归。**
- 验收时跑历史 bug 案例
- 修复一个 bug = 添加一个回归用例
- 回归失败不进入下一阶段
怎么做：回归测试 + 案例库联动，bug 修复 = 新增回归用例。

**7. 错误回滚记录。**
- 回滚事件也进 bug-cases
- 回滚原因 + 回滚步骤 + 影响范围
- 用于改进防御清单
怎么做：bug-cases 含"回滚案例"分类。

**8. 同类复发的强制 review。**
- 同类 bug 复发 = 防御失效，必须升级 review
- 防御失效的根因分析必须落档
- 防御清单升级后才能继续
怎么做：bug 复发触发强制 review + 防御升级。

**9. 知识库的去重与归档。**
- 同类案例 > 3 个 → 抽象为反模式
- 长期未引用的反模式 → 标记"待 review"
- 决策推翻 → 旧决策归档不删除
怎么做：定期 review 知识库，去重归档不删除历史。

**10. 跨项目的知识迁移。**
- 反模式抽象到通用方法论层（套 A 06）
- 套 B 的具体实现不污染套 A
- 跨项目复用反模式优先于项目级案例
怎么做：套 A 沉淀通用反模式，套 B 沉淀项目级案例。
