# 02 · 阶段化拆解

> 核心命题：怎么把"做一个原型"拆成可执行、可追踪、可验证的小块

---

## 原则

**1. 三层拆解结构。**
- Phase（阶段）→ 批次（Batch）→ 端点（Endpoint）
- 自上而下：业务域 → 相似度 → 页面体量
- 自下而上：端点组合成批次、批次组合成 Phase
怎么做：先按业务域分 Phase，再按规模/相似度分批次，最后按页面行数定端点数。

**2. Phase 按业务域划分。**
- 一个业务域 = 一个 Phase（不强求一对一，但尽量对齐）
- Phase 之间有依赖（上游产物 = 下游输入）
怎么做：用业务域清单做 Phase 划分的依据，避免按"人来切"。

**3. 批次按规模/相似度划分。**
- 同构模板 → 同一批次（一次扫描+改造可复用）
- 重型/特殊页 → 单独批次
- 每个 Phase 拆 2~4 个批次为宜
怎么做：每个 Phase 内按"轻量 / 中量 / 重量"三档分组。

**4. 端点按页面体量划分。**
- 轻量页（< 500 行）→ 1 端点
- 中型页（500~1500 行）→ 1~2 端点
- 重型页（> 1500 行）→ 2~3 端点
- 极重页（5+ tab）→ 4~5 端点（按 tab 拆）
怎么做：用页面行数 + tab 数做端点数决策。

**5. 批次间依赖明确。**
- 上游产物是下游输入
- 跨批次需要交接包（已完成/待办/决策/风险）
怎么做：每个批次计划含"上游输入 + 下游输出"两栏。

**6. 批次内部可并行。**
- 同批次内多个任务彼此独立、文件不重叠
- 可由多个执行者并行处理
- 并行任务通过知识库共享而非直接通信
怎么做：批次内任务明确"互斥资源 + 共享资源"。

**7. Phase 退出标准。**
- Phase 完成的标志 = 扫描 + 规划 + 改造 + 验收 全部完成
- 退出前必须生成验收报告
- 退出后才进入下一 Phase
怎么做：每个 Phase 都有硬性 checklist，未达标不退出。

**8. 跨阶段 = 架构基线锁定。**
- 跨阶段时打基线（git tag + 锁定文档）
- 锁定后只允许 bug 修复
- 大动作变更必须新建 Phase
怎么做：用 git tag + 锁定文档做"原型的最后 1 公里"。

**9. 拆解粒度的反模式。**
- ❌ 拆得太细：管理成本 > 收益
- ❌ 拆得太粗：风险集中、回滚困难
- ✅ 黄金区间：单批次 3~7 个同类任务
怎么做：批次规模偏离 3~7 时重新评估拆解。
